# KindEdit v1.0.0 软件开发规格说明

## 1. 产品定位

KindEdit 是一款轻量级文本类文件查看、编辑、格式化和结构浏览工具。当前重点支持 JSON，并扩展支持 TXT 纯文本文件、LOG 日志文件、XML 文件、Markdown 文件、Java 源码文件、Python 源码文件、JavaScript 源码文件和 SQL 脚本文件。

核心目标：

- 使用 Python 3.12 开发。
- 应用核心 GUI 仅使用 Python 标准库；如某类文档能力确需第三方运行依赖，必须在依赖文件、规格说明和打包说明中明确声明用途。
- GUI 使用 Python 标准库 `tkinter` 实现。
- 支持 Windows 和 macOS。
- 支持明暗主题。
- 支持 JSON 文本编辑、实时语法检查、语法渲染、树形结构查看、节点复制、文本格式化和压缩。
- 支持 TXT 纯文本文件编辑；TXT 不进行语法渲染，不显示结构树，仅显示单文本编辑窗口。
- 支持 LOG 日志文件编辑；LOG 不进行语法渲染，不显示结构树，仅显示单文本编辑窗口。
- 支持 XML 文本编辑、实时语法检查、语法渲染、树形结构查看、节点复制、文本格式化和压缩。
- 支持 Markdown 文本编辑、源码语法渲染和实时预览；Markdown 使用左右分栏，左侧显示渲染预览，右侧显示 Markdown 源文本。
- 支持 Java 源码文件编辑、基础语法检查和语法渲染；Java 当前不显示结构树，仅显示单文本编辑窗口。
- 支持 Python 源码文件编辑、语法检查和语法渲染；Python 当前不显示结构树，仅显示单文本编辑窗口。
- 支持 JavaScript 源码文件编辑、基础语法检查和语法渲染；JavaScript 当前不显示结构树，仅显示单文本编辑窗口。
- 支持 SQL 脚本文件编辑、基础语法检查和语法渲染；SQL 当前不显示结构树，仅显示单文本编辑窗口。
- 支持按文件类型切换视图布局，JSON 和 XML 使用左右分栏结构树，Markdown 使用左右分栏预览，TXT、LOG、Java、Python、JavaScript 和 SQL 使用单文本窗格。
- TXT 需要考虑大量内容或大文本性能，不能在每次按键时读取全文、解析全文或重建结构视图。
- LOG 需要考虑大量内容或大文本性能，不能在每次按键时读取全文、解析全文或重建结构视图。
- 支持大文本保护模式：打开 20MiB 及以上文件或编辑器内容达到约 5000000 字符后，文件打开、粘贴、自动保存和保存使用分块流程，避免 UI 线程长时间停顿。
- 大文本保护模式下暂停全量解析、语法渲染、结构树、Markdown 预览、全文格式化、全文压缩和未选区全文复制；普通小文本行为保持不变。
- 支持多页签编辑和会话恢复。

应用名称和主窗口标题：

- 应用名称：`KindEdit`
- 默认窗口标题：`KindEdit`
- 打开文件或切换页签后标题格式：`KindEdit - <页签标题或文件名>`

版本与品牌常量：

- `APP_TITLE = "KindEdit"`
- `LEGACY_APP_TITLES = ("Kind" "See", "JsonView")`
- `APP_VERSION_TEXT = "当前版本:v1.0.0"`
- `LATEST_VERSION_TEXT = "获取最新版本"`
- `BRAND_TEXT = "菜鸟驿站出品"`
- `LATEST_VERSION_URL = "https://www.cainiaoyizhan.com"`

## 2. 技术约束

开发语言：

- Python 3.12。

运行依赖：

- 核心 GUI 使用 Python 标准库。
- GUI 使用 `tkinter`、`tkinter.ttk`、`tkinter.font`、`tkinter.filedialog`。
- JSON 解析使用标准库 `json`。
- SQL 解析和格式化使用第三方运行依赖 `sqlglot`；依赖声明位于 `requirements.txt`，用途仅限 SQL 语法解析、错误检查和 SQL 美化。
- 文件拖拽打开使用第三方运行依赖 `tkinterdnd2` 提供 TkDND 扩展；依赖声明位于 `requirements.txt`，项目可从本地 `vendor/` 优先加载该依赖以避免修改系统 Python。macOS Tcl/Tk 9 运行时应使用项目本地兼容的 TkDND 2.9.5 运行库。
- 配置持久化使用标准库 `json`、`pathlib`、`os`。
- 主题检测可使用平台标准能力：
  - macOS：调用 `defaults read -g AppleInterfaceStyle`。
  - Windows：读取注册表 `Software\Microsoft\Windows\CurrentVersion\Themes\Personalize` 下的 `AppsUseLightTheme`。

可选能力：

- 打包脚本可使用 PyInstaller。PyInstaller 是打包工具，不是应用运行依赖，必须在文档和脚本中明确说明。

第三方依赖处置：

- `requirements.txt` 是源码运行的权威依赖入口，必须包含所有运行时第三方依赖。目前包括：
  - `sqlglot>=28.0.0`：用于 SQL 解析校验和格式化。
  - `tkinterdnd2>=0.4.3`：用于加载 TkDND 文件拖拽能力。
- 下载源码到新机器后，推荐在项目目录执行 `py -3.12 -m pip install -r requirements.txt`（Windows）或 `python3.12 -m pip install -r requirements.txt`（macOS/Linux）后再启动。
- 推荐使用虚拟环境隔离依赖：
  - Windows：`py -3.12 -m venv .venv`，然后执行 `.venv\Scripts\activate` 和 `python -m pip install -r requirements.txt`。
  - macOS/Linux：`python3.12 -m venv .venv`，然后执行 `source .venv/bin/activate` 和 `python -m pip install -r requirements.txt`。
- 项目允许携带本地 `vendor/` 目录作为离线或特定平台兼容依赖目录；应用启动时应优先把 `<项目根>/vendor` 加入 `sys.path`，再回退到 Python 环境已安装包。
- `vendor/` 仅用于运行时必须携带或平台兼容性特殊的依赖内容，不得放入构建缓存、`__pycache__`、日志或临时文件。
- `tkinterdnd2` 在不同平台携带不同 TkDND 二进制运行库。Windows 源码运行可直接使用 `pip install -r requirements.txt` 安装的 `tkinterdnd2`；如果源码包随附 `vendor/tkinterdnd2/`，应用也应能直接使用该本地依赖。
- macOS Homebrew Python 3.12 可能链接 Tcl/Tk 9；PyPI `tkinterdnd2` 当前自带的 macOS arm64 TkDND 2.9.3 与 Tcl/Tk 9 不兼容。源码包如面向该环境交付，应随附兼容 Tcl/Tk 9 的 `vendor/tkinterdnd2/tkdnd/osx-arm64-tcl9/`，应用应优先加载该目录。
- `sqlglot` 缺失时，SQL 功能必须降级到内置基础校验和基础格式化，应用不得启动失败。
- `tkinterdnd2` 或 TkDND 二进制运行库缺失或加载失败时，文件拖拽打开应静默降级；应用仍必须可通过打开文件对话框和快捷键打开文件。
- 启动脚本不应自动修改系统 Python 或全局安装依赖；需要安装依赖时，应由用户显式执行 `pip install -r requirements.txt` 或使用虚拟环境。
- 打包脚本必须把运行时依赖纳入产物。`tkinterdnd2` 的 TkDND 数据目录必须通过 PyInstaller `--add-data` 带入，避免发布到其它机器后拖拽能力缺失。

## 3. 配置与持久化

应用需要保存用户配置和页签会话。

配置目录：

- Windows：优先使用 `%APPDATA%\KindEdit`；如果不可用，使用 `%LOCALAPPDATA%\KindEdit`；仍不可用则使用用户主目录下的 `KindEdit`。
- macOS：`~/Library/Application Support/KindEdit`
- 其它平台：`${XDG_CONFIG_HOME}/KindEdit` 或 `~/.config/KindEdit`
- 从旧名称升级时，如果新配置目录没有 `settings.json`，允许读取上一应用名目录或旧 `JsonView` 配置目录中的 `settings.json` 作为兼容迁移来源；后续保存写入 `KindEdit` 配置目录。

配置文件：

- 路径：`<配置目录>/settings.json`

自动保存目录：

- 路径：`<配置目录>/tabs`

配置字段：

- `tree_font_size`：视图树字体大小。
- `text_font_size`：文本编辑器字体大小。
- `theme`：主题配置，取值为 `system`、`light`、`dark`。
- `text_content`：旧版本遗留字段，v2 多页签迁移时使用。
- `sync_display`：文本与视图是否同步显示。
- `occurrence_ignore_case`：选中高亮是否忽略大小写。
- `sql_uppercase_keywords`：SQL 格式化时是否大写关键字。
- `tabs`：页签元数据列表。
- `active_tab_id`：当前激活页签 ID。

默认值：

- 视图树字体大小：10。
- 文本编辑器字体大小：10。
- 页签标题字体大小：10。
- 默认主题：`system`。
- 默认开启同步显示。
- 默认关闭选中高亮忽略大小写。
- 默认开启 SQL 格式化大写关键字。

字体大小边界：

- 文本和视图字体大小必须限制在 8 到 28 之间。

页签元数据：

- `id`：页签唯一 ID，建议使用 UUID hex。
- `title`：页签标题。
- `file_path`：文件路径。为空表示未保存到外部文件。
- `autosave_path`：自动保存文件路径。
- `dirty`：是否已修改。
- `document_type`：文档类型 ID，例如 `json`、`text`、`log`、`xml`、`markdown`、`java`、`python`、`javascript`、`sql`。
- `document_type_locked`：文档类型是否由用户手动指定。为 `true` 时不再按内容自动切换类型。
- `large_content`：是否为大文本保护模式页签。
- `content_loaded`：大文本内容是否已加载到当前编辑器控件中。
- `large_source_path`：大文本来源路径。可以是外部文件路径，也可以是内部自动保存快照路径。
- `content_size`：大文本内容大小估计值，用于状态显示和会话恢复。

持久化要求：

- 保存配置失败不能影响应用使用。
- 保存自动恢复内容失败不能中断应用。
- 新建但未保存到外部文件的页签内容写入自动保存目录。
- 已保存到外部文件的页签恢复时优先读取外部文件路径。
- 大文本页签不得把完整内容写入配置文件；会话只保存元数据和来源路径。
- 大文本自动保存应写入自动保存快照文件，并采用分块写入，不能在 UI 线程一次性复制完整文本。
- 关闭应用时保存当前活动页签内容和页签元数据。

## 5. 主窗口与整体布局

窗口要求：

- 默认窗口尺寸：约 `1034x680`。
- 最小窗口尺寸：`880x560`。
- 初始打开时窗口应显示在屏幕中心。
- 主窗口背景应与当前主题一致。
- macOS 主窗口必须保留系统原生关闭、最小化和缩放按钮的可用状态，不得通过标题栏 style 适配把主窗口变为不可调整尺寸窗口。

主布局：

- 顶部第一行：菜单按钮栏。
- 顶部第二行：页签栏。
- 中部：左右分栏内容区。
- 左侧：结构树视图。JSON、XML 等层级文档类型显示结构树；TXT、LOG 等纯文本类型隐藏该区域。
- 右侧：文本编辑器。
- 底部：状态栏。

顶部菜单按钮：

- 顶部菜单按钮使用无边框图标按钮。
- 顶部菜单按钮栏背景应与页签栏背景区分；明亮主题下使用白色背景。
- `打开` 图标按钮：打开一个或多个文件。
- `保存` 图标按钮：保存当前页签。
- `设置` 图标按钮：打开设置窗口。
- `关于` 文字按钮：弹窗显示当前版本、获取最新版本地址和品牌信息。
- 顶部菜单按钮悬停时应显示高亮背景和细边框，鼠标移出后恢复普通无边框外观。
- 顶部菜单按钮的上下留白应保持紧凑，避免图标区域显得过高。
- `打开`、`保存`、`设置` 三个图标应视觉尺寸一致，不使用系统原生按钮边框。
- `打开`、`保存`、`设置` 三个图标应严格参考线框截图风格，使用同一套 Canvas 线框矢量绘制，统一约 28×28 视觉尺寸、统一 3px 左右描边、统一灰黑线条，不混用 emoji、彩色填充、系统图标和几何草图。
- `打开` 图标应表现为线框文件夹，`保存` 图标应表现为线框软盘，`设置` 图标应表现为线框齿轮；三个图标应视觉重量一致，且不绘制下方文字。
- 顶部菜单按钮悬停时使用普通指针，不使用手形光标。
- 顶部和树视图不提供 `类型` 菜单入口；文件类型检测和切换保留在文本编辑器右键菜单中。

鼠标样式：

- 超链接区域使用手形光标。
- 文本编辑器区域使用文本编辑光标。
- 除超链接和文本编辑器外，其它区域统一使用普通指针，包括顶部菜单按钮、页签标题、页签关闭按钮、页签新增按钮、树节点、右键菜单和普通状态栏文字。

分栏要求：

- 使用水平分栏。
- 左侧视图面板最小宽度约 280。
- 右侧文本面板最小宽度约 420。
- 首次启动后分栏位置约为窗口宽度的三分之一，但不小于 320。
- 分栏拖拽条应使用主题内的低对比颜色，不要突兀。

## 6. 页签系统

页签位于窗口顶部。

页签行为：

- 支持多个文本类文档页签，包括 JSON、TXT、LOG、XML、Markdown、Java、Python、JavaScript 和 SQL。
- 首次启动没有历史页签时创建一个“未命名”页签，文档类型默认为 TXT。
- 新建页签标题为 `未命名<N>`，未指定内容和文件类型时文档类型默认为 TXT。
- 新建未命名页签必须避开当前已存在的 `未命名<N>` 标题；恢复会话中已有 `未命名1` 时，再新增页签应从下一个可用编号开始。
- 页签标题已修改时追加 ` *`。
- 点击页签切换当前文档。
- 点击页签关闭按钮关闭页签。
- 在页签名称区域右键应弹出页签菜单，支持关闭当前文档、关闭其它文档、关闭右侧所有和关闭左侧所有；页签名称和背景绘制项应直接绑定辅助点击事件，不能只依赖页签栏全局坐标扫描；macOS、Windows 和不同 Tk 输入设备下的辅助点击事件都应被兼容处理；页签栏应把鼠标按钮事件先归一为“页签上下文菜单调用”语义，再通过 Tk 原生菜单在当前事件栈空闲后弹出，避免按下/释放阶段、Canvas 当前项、悬停提示窗口或平台菜单 grab 顺序导致菜单不显示；批量关闭时仍必须逐个遵循未保存确认规则，任意确认取消时中止后续关闭。
- 关闭未保存页签前必须弹出确认窗口，提供 `保存`、`不保存`、`取消` 三种选择；`取消` 中止关闭。
- 如果只剩一个页签，关闭操作等价于清空当前文本，并将文档类型重置为 TXT。
- 直接关闭整个窗口时，应按页签逐个确认所有未保存页签；任意页签选择 `取消` 时中止关闭整个窗口。
- 文件打开时，如果同一文件已经在某个页签中，则直接切换到该页签，不重复打开。

页签 UI：

- 页签栏整体无厚边框。
- 页签栏应使用集中式绘制与命中测试，避免原生按钮边框和控件重建导致的点击失效。
- 当前选中页签通过背景色和底部强调色线条表示。
- 非选中页签使用次要文字颜色。
- 页签包含标题和关闭按钮 `×`。
- 页签栏末尾有 `+` 按钮用于新建页签。
- 页签标题字体大小固定为 10 号。
- 页签标题最多显示 20 个字符，超过时截取前 17 个字符并在末尾追加 `...`；页签标题包含扩展名，截断规则也按包含扩展名的标题计算。
- 页签标题绘制必须为关闭按钮 `×` 预留固定空间；截断后的 `...` 不得与关闭按钮重叠。
- 鼠标悬停在页签标题上时，已保存文件显示完整文件路径；未保存页签显示完整页签标题。
- 页签悬停提示不得阻断页签右键菜单；当完整路径提示窗口覆盖页签名称区域时，在提示窗口上释放右键也必须弹出同一页签菜单。

## 6.1 文档类型与布局

应用通过文档类型策略决定解析、格式化、结构显示和布局模式。

文档类型字段：

- `type_id`：文档类型 ID。
- `display_name`：文件对话框显示名称。
- `default_extension`：默认保存扩展名。
- `file_dialog_patterns`：文件对话框筛选模式。
- `view_mode`：视图模式，取值包括 `split`、`preview`、`text`。
- `parse_on_change`：是否在文本变化后实时解析。
- `supports_format`：是否支持格式化。
- `supports_compact`：是否支持压缩。
- `empty_status`：空内容状态提示。

JSON 文档类型：

- `type_id = "json"`。
- `view_mode = "split"`。
- 使用左右分栏，左侧树视图，右侧文本编辑器。
- `parse_on_change = True`。
- 支持格式化和压缩。

TXT 文档类型：

- `type_id = "text"`。
- `view_mode = "text"`。
- 只显示单文本编辑窗口，不显示左侧树视图。
- `parse_on_change = False`。
- 不进行语法渲染。
- 不构建结构树。
- 不支持格式化和压缩，对应右键菜单项应禁用。
- 文本变化时只标记脏状态和延迟自动保存，不应每次按键读取全文或执行解析。

LOG 文档类型：

- `type_id = "log"`。
- `view_mode = "text"`。
- 只显示单文本编辑窗口，不显示左侧树视图。
- `parse_on_change = False`。
- 不进行语法渲染。
- 不构建结构树。
- 不支持格式化和压缩，对应右键菜单项应禁用。
- 文本变化时只标记脏状态和延迟自动保存，不应每次按键读取全文或执行解析。

XML 文档类型：

- `type_id = "xml"`。
- `view_mode = "split"`。
- 使用左右分栏，左侧树视图展示 XML 元素、属性和文本节点，右侧文本编辑器。
- `parse_on_change = True`。
- 支持格式化和压缩。
- 语法错误时显示行列和错误原因。

Markdown 文档类型：

- `type_id = "markdown"`。
- `view_mode = "preview"`。
- 使用左右分栏，左侧为只读渲染预览，右侧为 Markdown 源文本编辑器。
- `parse_on_change = True`。
- 支持根据 `.md`、`.markdown`、`.mdown` 扩展名识别文件类型。
- 支持标题、段落、引用、无序列表、有序列表、任务列表、分隔线、代码块、行内代码、粗体、斜体、链接、图片占位文本和表格的基础预览。
- 支持 Markdown 源文本语法渲染；预览控件由通用 UI 层负责渲染，文档类型模块不得直接操作 Tk 控件。
- 对未闭合围栏代码块显示语法错误；其它 Markdown 语法尽量宽容渲染，不因非严格写法阻断编辑。
- 当前不支持格式化和压缩，对应右键菜单项应禁用。
- 大文件语法检查和语法渲染需要有字符数上限，超过上限时不做全量检查，避免输入卡顿。

Java 文档类型：

- `type_id = "java"`。
- `view_mode = "text"`。
- 只显示单文本编辑窗口，不显示左侧树视图。
- `parse_on_change = True`。
- 进行基础语法检查，包括括号匹配、块注释闭合、字符串和字符字面量闭合。
- 不构建结构树。
- 当前不支持格式化和压缩，对应右键菜单项应禁用。
- 大文件语法检查需要有字符数上限，超过上限时不做全量检查，避免输入卡顿。

Python 文档类型：

- `type_id = "python"`。
- `view_mode = "text"`。
- 只显示单文本编辑窗口，不显示左侧树视图。
- `parse_on_change = True`。
- 使用 Python 标准库 `ast` 进行语法检查。
- 不构建结构树。
- 当前不支持格式化和压缩，对应右键菜单项应禁用。
- 大文件语法检查需要有字符数上限，超过上限时不做全量检查，避免输入卡顿。

JavaScript 文档类型：

- `type_id = "javascript"`。
- `view_mode = "text"`。
- 只显示单文本编辑窗口，不显示左侧树视图。
- `parse_on_change = True`。
- 进行基础语法检查，包括括号匹配、行注释、块注释、字符串和模板字符串闭合。
- 不构建结构树。
- 当前不支持格式化和压缩，对应右键菜单项应禁用。
- 大文件语法检查需要有字符数上限，超过上限时不做全量检查，避免输入卡顿。

SQL 文档类型：

- `type_id = "sql"`。
- `view_mode = "text"`。
- 只显示单文本编辑窗口，不显示左侧树视图。
- `parse_on_change = True`。
- 优先使用第三方运行依赖 `sqlglot` 进行 SQL 解析校验；未安装依赖时降级到内置基础校验，避免应用无法启动。
- 依赖可用时，SQL 校验能力应接近 JSON、XML、Python 等解析器驱动的校验模式；但 SQL 存在多方言差异，必须明确当前默认方言。
- 当前默认 SQL 方言为 `mysql`。
- 不构建结构树。
- 支持 SQL 格式化；依赖可用时使用 `sqlglot` 美化，依赖缺失时使用内置基础格式化；当前不支持压缩，对应压缩右键菜单项应禁用。
- SQL 格式化支持当前选中文本；存在选区时只替换选中内容，不改变选区之外的文本。
- SQL 格式化内容包含多个语句时，语句之间保留一个空行。
- SQL 格式化遵循设置项 `sql_uppercase_keywords`；开启时关键字大写，关闭时关键字小写，字符串和标识符内容不得被误改。
- 大文件语法检查需要有字符数上限，超过上限时不做全量检查，避免输入卡顿。

内容自动检测：

- 对未保存到外部文件的页签，应在明确的整段内容进入文本框时根据文本内容自动检测文档类型。
- 自动检测不应在用户普通逐字编辑时反复触发，避免编辑过程中类型频繁变化。
- 自动检测触发时机包括：
  - 文本框原内容为空或只有空白符，随后变为非空内容；此时应强制重新检测，并解除旧的手动指定状态。
  - 用户全选文本后粘贴新内容，等价于整段内容被替换；此时应强制重新检测，并解除旧的手动指定状态。
- 自动检测成功后，应立即把当前页签切换为检测出的文档类型，并刷新布局、语法检查和语法渲染。
- 内容检测不应只依赖第一行或注册顺序；文档类型策略应基于整段小文本样本返回置信度，注册表统一选择最高置信度类型。
- JSON、XML 等可低成本完整解析的结构化格式应优先使用解析成功作为强信号；代码、Markdown、LOG 等文本格式应使用多特征规则，避免单个孤立关键字造成误判。
- 文本编辑器右键菜单中的 `类型` 二级菜单，应把自动检测出的当前类型显示为选中状态。
- 用户通过文本编辑器右键菜单手动切换类型后，该页签进入手动指定状态；普通逐字编辑不再被内容自动检测覆盖，但后续空白到非空或整段替换仍应重新检测。
- 自动检测无法识别内容时，未保存页签应回到 TXT 文档类型。
- 打开真实文件时优先按文件扩展名识别类型，不受内容自动检测影响。
- 保存为真实文件后，类型按保存路径扩展名重新识别。
- 内容自动检测应有性能保护，不能在大文本上做高成本解析。

## 6.2 语法渲染系统

语法渲染通过独立的 `syntax` 模块管理。

通用规则：

- 语法渲染器只负责把文本识别为语义 token。
- 文本编辑器负责根据 token 和当前主题应用颜色。
- 文档类型模块不得直接操作 Tk 文本控件颜色。
- 语法颜色由主题模块统一定义。
- 语法渲染必须有性能保护，避免大文件每次按键全量高成本渲染。

语法 token 字段：

- `kind`：语法类型，例如 `key`、`string`、`number`、`literal`、`punctuation`。
- `start`：起始字符偏移。
- `end`：结束字符偏移。

JSON 语法渲染：

- 对对象键名使用 key 样式。
- 对字符串值使用 string 样式。
- 对数字使用 number 样式。
- 对 `true`、`false`、`null` 使用 literal 样式。
- 对 `{}`、`[]`、`:`、`,` 使用 punctuation 样式。
- JSON 语法渲染不要求 JSON 必须完全合法，输入非法 JSON 时仍应尽量渲染可识别 token。

TXT 语法渲染：

- TXT 不进行语法渲染。
- TXT 使用空高亮器，不产生 token。

LOG 语法渲染：

- LOG 不进行语法渲染。
- LOG 使用空高亮器，不产生 token。

XML 语法渲染：

- 对元素名和属性名使用 key 样式。
- 对属性值使用 string 样式。
- 对注释、CDATA 等特殊内容使用 literal 样式。
- 对 `<`、`>`、`/`、`=`、`?`、`!` 等标签标点使用 punctuation 样式。
- XML 语法渲染不要求 XML 必须完全合法，输入非法 XML 时仍应尽量渲染可识别 token。

Java 语法渲染：

- 对 Java 关键字使用 key 样式。
- 对字符串和字符字面量使用 string 样式。
- 对数字字面量使用 number 样式。
- 对 `true`、`false`、`null`、注解、行注释和块注释使用 literal 样式。
- 对括号、分号、逗号、运算符等符号使用 punctuation 样式。
- Java 语法渲染不要求 Java 代码必须完全合法，输入非法 Java 时仍应尽量渲染可识别 token。

Python 语法渲染：

- 对 Python 关键字使用 key 样式。
- 对字符串使用 string 样式。
- 对数字字面量使用 number 样式。
- 对 `True`、`False`、`None`、注释使用 literal 样式。
- 对括号、冒号、逗号、运算符等符号使用 punctuation 样式。
- Python 语法渲染不要求 Python 代码必须完全合法，输入非法 Python 时仍应尽量渲染可识别 token。

JavaScript 语法渲染：

- 对 JavaScript 关键字使用 key 样式。
- 对字符串和模板字符串使用 string 样式。
- 对数字字面量使用 number 样式。
- 对 `true`、`false`、`null`、`undefined`、行注释和块注释使用 literal 样式。
- 对括号、分号、逗号、运算符等符号使用 punctuation 样式。
- JavaScript 语法渲染不要求 JavaScript 代码必须完全合法，输入非法 JavaScript 时仍应尽量渲染可识别 token。

SQL 语法渲染：

- 对 SQL 关键字使用 key 样式。
- 对字符串、双引号标识符、反引号标识符和方括号标识符使用 string 样式。
- 对数字字面量使用 number 样式。
- 对 `true`、`false`、`null`、行注释和块注释使用 literal 样式。
- 对括号、分号、逗号、运算符等符号使用 punctuation 样式。
- SQL 语法渲染不要求 SQL 语句必须完全合法，输入非法 SQL 时仍应尽量渲染可识别 token。

Markdown 语法渲染：

- 对标题标记、列表标记和粗体内容使用 key 样式。
- 对行内代码和围栏代码块内容使用 string 样式。
- 对引用标记、链接、图片、斜体内容和代码块围栏使用 literal 样式。
- 对分隔线和表格分隔行使用 punctuation 样式。
- Markdown 语法渲染不要求 Markdown 必须完全合法，输入非法 Markdown 时仍应尽量渲染可识别 token。

## 7. 文本编辑器

文本编辑区功能：

- 用于输入、粘贴和编辑文本内容，适用于 JSON、TXT 和后续新增文本类文件。
- 支持水平和垂直滚动。
- 普通鼠标滚轮滚动幅度应保持可感知效率；文本编辑器、Markdown 预览、JSON/XML 树视图等所有主内容区域每次滚轮事件统一滚动 6 个单位。
- 支持成熟编辑器粒度的快捷键撤销和重做：macOS 使用 `Command+Z` / `Command+Shift+Z`，Windows/Linux 使用 `Ctrl+Z` / `Ctrl+Y`，并兼容 `Ctrl+Shift+Z`。
- 撤销栈由文本编辑器统一管理，不依赖 Tk 默认自动分组；连续普通单词输入可按短暂停顿合并为一步，空格、标点、回车、Tab、删除、粘贴、剪切、格式化、压缩、清空和列编辑等明确编辑动作必须形成清晰撤销边界。
- 程序化替换内容时应作为一个完整撤销步骤；打开文件、切换页签或恢复会话后不得把载入前的历史编辑带入当前页签撤销栈。
- 不自动换行，保持原始横向结构。
- 左侧显示行号。
- 行号栏宽度根据当前总行数动态计算：
  - 1 到 9 行时只保留个位数所需宽度。
  - 10 到 99 行时保留两位数所需宽度。
  - 100 行以上按实际位数增长。
- 行号颜色使用主题的次要文字色。

列编辑：

- 所有文档类型的文本编辑窗口都支持矩形列编辑。
- Windows/Linux 使用 `Alt + 鼠标拖动` 开始列编辑；macOS 使用 `Option + 鼠标拖动`。
- `Mod1`、`Mod2` 等 Tk 泛化修饰键绑定只能作为 macOS Option 兼容路径使用，不得在 Windows/Linux 普通文本框点击路径中注册，避免 NumLock 或系统状态被误判为列编辑。
- 普通鼠标单击、双击和无修饰键拖动必须保留 Tk 文本控件原生插入光标、选词和拖拽选区行为，不得启动列编辑或残留列编辑竖线。
- 列编辑按字符边界计算列位置；在两个字符之间垂直拖动时应显示一条细竖线，不应高亮竖线前后任一字符。
- 列编辑选区覆盖多行同一列范围，输入普通字符时应在每一行相同列位置插入内容；短行应补空格后再插入。
- 列编辑状态下支持 `BackSpace` 和 `Delete` 对每一行同列位置删除字符，按 `Esc` 退出列编辑。

大文本保护模式：

- 文件大小达到 20MiB 及以上时，打开文件不允许使用一次性 `read_text()` 读入完整内容，应创建大文本页签并在文本控件中分块加载。
- 编辑器内容达到约 5000000 字符时，该页签进入大文本保护模式。
- 大文本保护模式下，页签布局强制使用单文本窗格，不显示结构树或 Markdown 预览。
- 大文本保护模式下，不执行全量内容自动检测、全量解析、结构树构建、语法渲染、Markdown 预览、全文格式化和全文压缩。
- 大文本保护模式下，普通按键编辑只更新脏状态、标题、内容大小和延迟保存任务，不得每次按键读取完整文本。
- 大文本保护模式下，自动保存和手动保存必须分块读取文本控件内容并写入文件。保存过程中状态栏应显示进度，UI 仍可响应。
- 大文本保护模式下，页签切换或关闭前如存在未保存修改，应先保存内部快照，再释放当前编辑器内容；会话恢复时从快照或外部文件重新分块加载。
- 大文本保护模式下，不允许未选中范围时复制全文；选区过大时应取消复制并在状态栏提示，避免把 200M 文本一次性放入剪贴板造成卡顿。
- 从剪贴板粘贴达到大文本阈值的内容时，应由应用拦截默认粘贴流程并分块插入。
- 普通小文本路径不进入该模式，继续保持原有解析、语法渲染、结构视图、格式化、压缩、复制、粘贴和自动检测行为。

相同内容高亮：

- 所有文档类型的文本编辑窗口都支持相同内容高亮。
- 用户双击选中内容，或通过鼠标拖拽、键盘等方式选中文本后，应在当前文本编辑窗口中高亮所有完全相同的内容。
- 普通选中文本时，当前选中内容替换已有相同内容高亮查询组。
- 按住追加选择修饰键选中文本时，当前选中内容追加到已有相同内容高亮查询组；Windows/Linux 使用 Ctrl，macOS 使用 Command，macOS 下 Ctrl 也可兼容识别但不作为主要操作方式。
- 按住追加选择修饰键双击选中文本时，也应把双击选中的词追加到已有相同内容高亮查询组。
- Windows 下按住 Ctrl 通过鼠标拖动或双击完成选择时，应可靠追加高亮查询组，不得被 Tk 平台事件差异误判为普通选择。
- Windows 下 Ctrl + 鼠标拖动和 Ctrl + 双击应由文本编辑器显式维护临时真实选区，然后把选中文本追加到高亮查询组；不能依赖 Tk Text 默认 Ctrl 鼠标 class binding。
- 多个被选中的不同内容应分别高亮其所有相同内容。
- 相同内容使用相同高亮颜色，不同内容使用不同高亮颜色。
- 追加到高亮查询组的内容必须由应用状态持久保存；松开 Ctrl/Command、Tk 当前真实选区只保留最后一次选择，或鼠标释放后选区显示变化，都不得清除之前追加的高亮查询词。
- 释放 Ctrl/Command 等追加选择修饰键时，即使当前仍有最后一次真实选区，也只能保留并重绘已有高亮查询组，不得把该选区当作普通选择替换查询组。
- 鼠标释放已经捕获到待追加选中文本后，后续 Ctrl/Command 释放不得取消或覆盖这次追加任务。
- 保留高亮查询组的刷新路径不得读取当前 Tk 真实选区，只能基于应用状态中已有的高亮查询词重绘。
- 为避免颜色和性能失控，当前最多保留 6 个相同内容高亮查询词；超过后保留最近选择的查询词。
- 设置项“选中高亮忽略大小写”关闭时，大小写不同的文本视为不同内容。
- 设置项“选中高亮忽略大小写”开启时，大小写不同但字母相同的文本视为相同内容。
- 相同内容高亮只作用于文本编辑窗口，不作用于 JSON、XML 等左侧结构树视图。
- 未选中文本、选中文本为空白、选中文本过长时，不触发相同内容高亮。
- 相同内容高亮应在鼠标释放后的空闲回调或键盘选区变化时捕获当前选中文本，再进行防抖高亮；不得等防抖回调执行时才读取 `sel.first/sel.last`，避免 Tk 内置选区处理尚未完成或后续释放修饰键导致追加高亮丢失。
- 为避免大文本卡顿，相同内容高亮必须有防抖和上限保护。
- 当前建议限制：
  - 文本总长度超过 500000 字符时不执行全局相同内容高亮。
  - 文本总长度超过 500000 字符时，仍应在当前可见区域及其附近范围内执行相同内容高亮，用户滚动到对应内容时能看到匹配高亮。
  - 选中文本超过 200 字符时不执行相同内容高亮。
  - 最多高亮 2000 处匹配。
  - 大文本可见区域高亮最多处理 300 处匹配。

文本字体：

- macOS：`Menlo`
- Windows：`Consolas`
- 字体大小可通过 `Ctrl + 鼠标滚轮` 调整。
- macOS 也可使用同样的 Ctrl 滚轮逻辑。
- 字体大小调整后保存到配置。

文本编辑器右键菜单：

- `格式化`
- `压缩`
- 分割线
- `类型`
  - 已注册文档类型扩展名列表，例如 `.json`、`.txt`、`.log`、`.xml`、`.md`、`.java`、`.py`、`.js`、`.sql`
- 分割线
- `同步`：开启时显示为选中状态，关闭时不显示选中状态；单文本窗口类型应置灰不可用。
- 分割线
- `复制`
- `粘贴`
- 分割线
- `清空`

右键菜单行为：

- 菜单应轻量、响应迅速。
- 菜单项在鼠标悬停时应显示高亮背景。
- 禁用菜单项应明显比普通菜单项更灰，并且鼠标悬停在禁用菜单项上不应显示高亮效果。
- 带二级菜单的菜单项在鼠标悬停时应自动展开子菜单。
- 右键菜单应优先使用 Tk 原生 `Menu` 组件承载级联菜单、悬停、外部点击关闭和 Esc 关闭等基础行为。
- 菜单项左侧应使用原生 checkbutton 标记列承载选中状态，不能影响各菜单项正文左侧对齐。
- 右键菜单只能在点击叶子菜单项、点击菜单区域外、按下 `Esc` 或打开新的右键菜单时关闭；在一级和二级菜单之间移动光标不得关闭菜单。
- `格式化`：如果当前文档类型支持格式化且文本合法，按当前文档类型输出可读格式；大文本保护模式下禁用。
- `压缩`：如果当前文档类型支持压缩且文本合法，按当前文档类型输出紧凑格式；大文本保护模式下禁用。
- `类型`：光标滑过后展开二级菜单；点击具体扩展名时把当前页签切换为指定文档类型，立即按目标类型刷新布局、解析结果和语法渲染，不修改文本内容，并标记为用户手动指定类型。
- `同步`：切换文本与结构视图的同步状态；只影响左右窗口选中和定位联动，不得影响文本窗口光标显示和编辑能力。
- `复制`：如果存在选中文本，则复制选中部分；否则复制全部文本。大文本保护模式下不得复制全文，选区过大时也应拒绝复制。
- `粘贴`：从系统剪贴板读取文本，替换当前选区或插入到光标位置，然后立即解析；达到大文本阈值时分块粘贴并进入大文本保护模式。
- `清空`：清空编辑器、清空视图数据并更新状态。

实时解析：

- 文本变化后需要延迟解析，推荐防抖时间约 140ms；内容类型自动检测只在空白到非空或整段替换时触发。
- 文本变化后需要延迟自动保存，推荐防抖时间约 350ms。
- 大文本保护模式下自动保存使用更长防抖和分块快照流程，避免每次输入后立刻复制完整内容。
- JSON、XML 等实时解析类型合法时，状态栏不显示语法成功提示。
- JSON、XML 等实时解析类型非法时，状态栏左侧显示错误信息：`语法错误：第 <行> 行，第 <列> 列，<错误原因>`。
- 文本为空时，视图清空，状态提示由当前文档类型决定，例如 `请输入 JSON 内容`、`请输入 XML 内容`。

## 8. JSON 树形视图

树形视图用于展示 JSON 结构。

根节点：

- 根节点固定命名为 `Root`。
- 根节点默认展开。

节点类型：

- 对象：`dict`
- 数组：`list`
- 普通字段：字符串、数字、布尔值、null 等。

节点显示文本：

- 对象节点：`<键名>: {<成员数量>}`
- 数组节点：`<键名>: [<元素数量>]`
- 普通字段：`<键名>: <值文本>`
- 根对象：`Root {<成员数量>}`
- 根数组：`Root [<元素数量>]`

值文本规则：

- 字符串直接显示字符串值，不额外加引号。
- `true`、`false`、`null` 使用 JSON 小写形式显示。
- 数字使用字符串形式显示。

默认展开规则：

- 根节点展开。
- 根节点的第一层子节点默认显示。
- 第一层子节点本身默认折叠，即数组或对象内部不继续默认展开。

树连接线：

- 父子节点之间用细的点画线连接。
- 点画线应从父节点类型图标底部中点引出，再连接到子节点类型图标左侧附近，避免连接到展开/折叠的 `+`、`−` 按钮。
- 点画线不能压过 `+`、`−` 按钮区域。
- 连接线颜色使用主题的低对比线条色。

节点图标：

- 对象图标符号：`{` 和 `}`
- 数组图标符号：`[` 和 `]`
- 普通值图标符号：`‹` 和 `›`
- 三种图标应按左右两个符号分开绘制，中间间距使用约八分之一空格宽度，不能用完整空格撑宽。
- 左右两个符号应以图标中心为基准放入对称槽位，保证图标左右留白视觉对称。
- 三种图标按 `{}`、`[]`、`‹›` 加压缩间距后的最大宽度统一占位，且宽度至少等于图标高度，让图标尽量趋近正方形并保证同层节点名称起始位置对齐。
- 三种图标外框高度必须统一，避免不同字形视觉高度不一致。
- 统一图标宽度只保留极少量左右内边距，总横向内边距约 1px，让字符两侧到边框仅有可辨识空隙，避免出现过大的横向空白。
- 三种图标分别使用浅色背景区分类型：对象浅红、数组浅黄、普通值浅绿；深浅顺序为浅红最深、浅黄次之、浅绿最淡，整体不能过重。

交互：

- 单击节点：选中节点。
- 单击 `+` 或 `−` 区域：展开或折叠当前节点。
- 双击有子节点的节点：展开或折叠当前节点。
- 点击空白区域：取消选中。
- 右键空白区域：取消选中并显示菜单。
- 支持鼠标滚轮垂直滚动。
- 支持 Shift + 鼠标滚轮横向滚动。
- 支持 Ctrl + 鼠标滚轮调整树视图字体大小。
- 支持 `Ctrl+C` 和 macOS `Command+C` 复制当前选中节点，复制内容应与右键菜单“复制”一致。

## 9. 树视图右键菜单

树视图右键菜单项：

- `展开所有`
- `展开此项`
- 分割线
- `折叠所有`
- `折叠此项`
- 分割线
- `复制`
- `复制值`
- `复制路径`

可用性规则：

- 没有合法 JSON 树时：
  - `展开所有` 禁用。
  - `折叠所有` 禁用。
  - 其它依赖选中节点的菜单项禁用。
- 没有选中节点时：
  - `展开所有` 根据全局是否还有可展开节点决定是否可用。
  - `折叠所有` 根据全局是否还有可折叠节点决定是否可用。
  - `展开此项`、`折叠此项`、`复制`、`复制值`、`复制路径` 禁用。
- 当前所有层级都已展开时：
  - `展开所有` 禁用。
- 当前没有任何可折叠节点时：
  - `折叠所有` 禁用。
- 当前选中节点没有子节点，或该节点及其所有子层级都已展开时：
  - `展开此项` 禁用。
- 当前选中节点没有子节点，或该节点当前未展开时：
  - `折叠此项` 禁用。

性能要求：

- 节点展开、折叠后应立即更新节点自己的 `can_expand_item`、`can_collapse_item` 状态，并向祖先节点刷新状态。
- 全局 `can_expand_all`、`can_collapse_all` 应在树结构变化、展开、折叠、选中变化后更新。
- 右键菜单打开时只读取已经维护好的状态，不应临时遍历大树做重计算。
- 右键菜单命令执行后应立即关闭菜单并刷新界面，用户感知上应和直接点击 `+`、`−` 一样轻快。

复制行为：

- `复制`：
  - 根节点：复制完整 JSON 值。对象和数组使用 `ensure_ascii=False, indent=2` 格式化。
  - 对象属性节点：复制 `"key": value`。
  - 数组元素节点：复制 `index: value`。
  - 普通值按 JSON 可读形式输出。
- `复制值`：
  - 对象或数组：复制格式化后的 JSON 值。
  - 字符串：复制字符串本身。
  - 布尔值和 null：复制 `true`、`false`、`null`。
  - 数字：复制数字字符串。
- `复制路径`：
  - 根节点路径：`Root`
  - 对象属性：使用点路径，例如 `Root.user.name`
- 数组元素：使用索引路径，例如 `Root.items[0].name`

## 8.1 XML 树形视图

树形视图用于展示 XML 层级结构。

根节点：

- 根节点固定命名为 `Root`，显示根元素名称，例如 `Root <root>`。
- 根节点默认展开。

节点类型：

- 元素节点：XML Element，使用对象图标。
- 属性节点：以 `@属性名` 显示，使用普通字段图标。
- 文本节点：以 `#text` 显示，使用普通字段图标。

节点显示文本：

- 元素节点：`<节点名>: <元素名> {<子项数量>}`。
- 属性节点：`@<属性名>: <属性值>`。
- 文本节点：`#text: <文本摘要>`。

复制行为：

- `复制` 和 `复制值`：
  - 元素节点复制格式化后的 XML 片段。
  - 属性和文本节点复制其字符串值。
- `复制路径`：
  - 根节点路径：`Root`
  - 元素、属性、文本节点使用点路径，例如 `Root.book.@id`、`Root.book.title.#text`。

## 10. 文本与视图同步显示

文本编辑器右键菜单提供“同步”开关；树视图右键菜单不显示“同步”。

开启同步显示时：

- 在文本编辑器中移动光标，应定位到最匹配的 JSON 节点。
- 如果对应节点在折叠结构内部，应自动展开父节点并滚动到该节点附近。
- 在树视图中单击节点，应只移动文本编辑器输入光标，不应先选中或高亮右侧 value；根节点把输入光标移动到顶层 `{` 或 `[` 的右侧，对象属性节点把输入光标移动到 key 开始双引号的右侧，数组元素等没有 key 的节点移动到该节点值起点。
- 同步不得改变文本编辑器的可编辑状态，不得隐藏或破坏插入光标。
- 文本编辑器区域的鼠标光标应始终保持文本编辑光标，不应因右键菜单或同步切换残留为指针光标。
- 文本编辑器鼠标光标应由文本控件自身固定配置，不应在 `<Motion>` 这类高频鼠标移动事件中反复 `configure(cursor=...)`；macOS 使用 `ibeam`，Windows 和其它 Tk 平台使用 `xterm`。
- 窗口初次显示、页签切换、右键菜单弹出或关闭后，可设置一次性光标刷新标记；文本区后续首次进入或移动时只允许执行一次强制刷新，刷新后立即清除标记，避免鼠标移动过程中持续闪烁。
- TXT、LOG、Java、Python、JavaScript、SQL 等单文本窗口类型不提供可用的同步切换。
- Markdown 预览分栏不使用结构树路径索引，当前不提供可用的同步切换。

位置索引：

- 需要基于原始 JSON 文本建立路径到字符偏移的索引。
- 索引不能改变用户原始文本格式。
- 需要支持对象、数组、字符串、数字、布尔值和 null。
- 对象路径使用属性名。
- 数组路径使用字符串形式的索引。
- 解析失败时返回空索引。

文本到树同步高亮规则：

- 字符串：高亮完整字符串，包括引号。
- `true`、`false`、`null`：高亮完整字面量。
- 数字：高亮完整数字。
- 对象或数组：高亮起始 `{` 或 `[`。

树到文本定位规则：

- 树节点单击不选中右侧 value，不触发临时 value 高亮。
- 根节点定位到顶层 `{` 或 `[` 之后。
- 对象属性节点定位到属性 key 开始双引号之后。
- 数组元素或其它没有 key 的节点定位到对应值起点。

关闭同步显示时：

- 文本光标移动不再影响树视图。
- 树节点选中不再影响文本编辑器。

## 11. 状态栏

状态栏位于窗口底部。

左侧：

- 显示实时状态或错误信息。
- JSON 语法正确时不显示成功信息。
- JSON 语法错误时显示错误位置。
- 临时操作成功信息可短暂显示，例如 `已保存`、`已打开：<文件名>`、`已新增页签`。

视觉：

- 状态栏高度约 21，为原高度约 3/4。
- 文字应垂直居中。
- 状态栏只显示左侧实时状态或错误信息，不显示版本、获取最新版本和品牌固定文案。
- 状态栏字体应适配压缩后的高度，默认使用 8 号字体。

## 11.1 设置窗口

点击顶部菜单按钮栏中的 `设置` 后弹出设置窗口。

设置项：

- `选中高亮忽略大小写`：复选框。勾选后，相同内容高亮匹配时忽略大小写；未勾选时严格区分大小写。
- `SQL 格式化大写关键字`：复选框。勾选后 SQL 格式化输出大写关键字；未勾选时 SQL 格式化输出小写关键字。
- `当前主题`：只显示标签文本，不额外显示 `明亮`、`暗色` 等当前值文案。
- 主题切换控件与 `当前主题：` 位于同一行末尾，显示无边框的一体式 `明` / `暗` 单选区域；点击该区域任意位置都在明亮和暗色主题之间切换，并写入配置。
- `重置`：按钮。位于左下角，与 `关闭` 按钮同一行；点击后重置字体大小、主题、同步显示、选中高亮忽略大小写和 SQL 格式化大写关键字等设置项，不清空页签内容。

行为：

- 设置窗口颜色应跟随当前主题。
- 设置窗口标题栏应和主窗口标题栏一样尽量跟随当前明暗主题；不支持的平台或 Tk 版本可降级。
- 设置窗口按钮和主题分段单选控件颜色应跟随应用主题；暗色主题下背景和文字必须有足够对比度。
- 修改 `选中高亮忽略大小写` 后，应立即影响当前文本编辑窗口中的相同内容高亮。
- 修改 `SQL 格式化大写关键字` 后，应立即影响后续 SQL 格式化命令。
- 主题切换和设置重置入口应位于设置窗口中，状态栏不提供这些入口。

## 12. 文件能力

打开文件：

- 支持通过文件对话框打开一个或多个文本类文件，默认筛选应为所有已支持的文本文件类型，而不是只默认 JSON。
- 文件对话框筛选：
  - `所有文本文件`
  - `JSON 文件 (*.json)`
  - `TXT 文件 (*.txt)`
  - `LOG 文件 (*.log)`
  - `XML 文件 (*.xml)`
  - `Java 文件 (*.java)`
  - `Python 文件 (*.py *.pyw)`
  - `JavaScript 文件 (*.js *.mjs *.cjs)`
  - `SQL 文件 (*.sql)`
  - `所有文件 (*.*)`
- 打开失败时在状态栏显示 `打开失败：<异常信息>`。
- 打开成功后创建或切换到对应页签。
- 文件内容按 UTF-8 读取。
- 打开 `.json` 文件时使用 JSON 文档类型。
- 打开 `.txt` 文件时使用 TXT 文档类型。
- 打开 `.log` 文件时使用 LOG 文档类型。
- 打开 `.xml` 文件时使用 XML 文档类型。
- 打开 `.java` 文件时使用 Java 文档类型。
- 打开 `.py`、`.pyw` 文件时使用 Python 文档类型。
- 打开 `.js`、`.mjs`、`.cjs` 文件时使用 JavaScript 文档类型。
- 打开 `.sql` 文件时使用 SQL 文档类型。
- 无法识别扩展名时默认按 JSON 文档类型处理。

保存文件：

- 快捷键保存当前页签。
- 如果当前页签已有 `file_path`，直接写回原文件。
- 如果没有 `file_path`，弹出保存文件对话框。
- 默认扩展名根据当前文档类型决定，JSON 为 `.json`，TXT 为 `.txt`，LOG 为 `.log`，XML 为 `.xml`，Java 为 `.java`，Python 为 `.py`，JavaScript 为 `.js`，SQL 为 `.sql`。
- 用户保存未输入扩展名时，自动补齐当前文档类型的默认扩展名；保存对话框的首选筛选也应优先显示当前文档类型。
- 保存失败时在状态栏显示 `保存失败：<异常信息>`。
- 保存成功后：
  - 页签 `dirty=false`。
  - 页签标题改为文件名。
  - 窗口标题改为 `KindEdit - <文件名>`。
  - 清理旧的内部自动保存文件。

文件拖拽：

- 应用启动时使用普通 `tk.Tk()` 创建根窗口，再按当前平台和 Tcl/Tk 版本把项目本地 TkDND 目录加入 Tk `auto_path`，最后递归注册拖放目标。
- 如果运行环境有 TkDND，允许拖拽文本类型文件到窗口、页签栏、工具栏、结构树、预览区或文本编辑区打开。
- Drop 回调必须通过 TkDND 专用事件数据 `%D` 获取文件路径，不能依赖普通 Tkinter 事件对象自动提供文件路径。
- 拖拽打开不按扩展名限制；只要是普通文件、可按常见文本编码读取且内容不像二进制，就允许打开。
- 拖拽打开支持 UTF-8、UTF-16 和 GB18030 等常见文本编码；目录、二进制文件和不可读取文件应跳过。
- 未知扩展名文本文件打开后按现有文档类型检测规则处理，无法按扩展名识别时使用默认文档类型。
- macOS 下可注册 `::tk::mac::OpenDocument` 作为系统打开文档入口，用于接收系统传入的文件路径并按同一文本文件过滤规则打开。
- TkDND 不可用时不报错，继续通过打开文件对话框和快捷键提供文件打开能力。

## 16. JSON 树构建与状态维护

树节点数据结构字段：

- `label`
- `value`
- `depth`
- `parent`
- `expanded`
- `can_expand_item`
- `can_collapse_item`
- `children`

节点属性：

- `has_children`：是否有子节点。
- `kind`：`object`、`array`、`field`。
- `display_text`：用于树视图显示的文本。

构建规则：

- 根节点 label 为 `Root`，depth 为 0，默认展开。
- 对象按原始 JSON 对象遍历顺序创建子节点。
- 数组按索引顺序创建子节点，label 为索引字符串。
- 所有子节点递归创建。
- 根节点的直接子节点默认折叠。

状态刷新：

- `can_collapse_item = has_children and expanded`
- `can_expand_item = has_children and (not expanded or any(child.can_expand_item))`
- 操作某个节点后，刷新该节点及祖先状态。

## 18. 打包脚本

需要提供 macOS 和 Windows 打包脚本。

macOS：

- 文件名建议：`build_macos.command`
- 使用 PyInstaller 打包为 `.app`。
- 命令形态：`python3.12 -m PyInstaller --noconfirm --windowed --name KindEdit --paths <项目根>/vendor --add-data <项目根>/vendor/tkinterdnd2:vendor/tkinterdnd2 kindedit.py`
- 打包产物：`dist/KindEdit.app`

Windows：

- 文件名建议：`build_windows.bat`
- 使用 PyInstaller 打包为 `.exe`。
- 命令形态：`py -3.12 -m PyInstaller --noconfirm --windowed --name KindEdit --paths <项目根>\vendor --add-data <项目根>\vendor\tkinterdnd2;vendor\tkinterdnd2 kindedit.py`
- 打包产物：`dist\KindEdit.exe`

说明：

- PyInstaller 是构建阶段第三方工具。
- 应用源码运行时不得依赖 PyInstaller。
- 打包前必须先安装 `requirements.txt` 中的运行依赖；打包产物必须包含 `tkinterdnd2` 及 TkDND 二进制数据目录。
- 如果没有随源码携带 `vendor/`，打包脚本应使用 Python 环境中已安装的 `tkinterdnd2`，或先生成本地 `vendor/` 后再打包。

## 19. 错误处理原则

- 配置读取失败：静默使用默认配置。
- 配置保存失败：静默忽略，不影响主功能。
- 自动保存失败：静默忽略，不影响编辑。
- 文件打开失败：状态栏显示错误。
- 文件保存失败：状态栏显示错误。
- JSON、XML 解析失败：状态栏显示行列和错误原因。
- 主题标题栏设置失败：静默降级。
- TkDND 不可用：静默降级。
- 剪贴板读取失败：忽略当前操作。

## 20. 性能要求

输入体验：

- 用户输入或粘贴 JSON、XML 等层级格式时不能每个按键立即重建大型树，应使用短防抖。
- 防抖后解析合法 JSON、XML，并提前准备树视图数据。
- 切换到视图窗口时不应再出现明显卡顿。

右键菜单体验：

- 菜单打开和命令触发应轻快。
- 菜单项可用状态应依赖已维护状态，不应在打开菜单时做大规模计算。

树视图体验：

- 点击 `+`、`−` 展开折叠应立即反馈。
- 右键菜单的展开折叠应与直接点击一样迅速。
- 展开、折叠、选中后只做必要重绘。

文本与树同步：

- 光标移动同步应做短延迟或轻量处理，避免每次移动都阻塞 UI。
- 节点路径索引可在 JSON 成功解析后构建并缓存；XML 当前仅构建结构树，不要求文本光标到 XML 节点的精确路径同步。

## 23. 给 AI 智能体的实现提示

实现时应优先保证以下行为完全正确：

- 优先使用标准库；如确需运行时第三方依赖，必须写入 `requirements.txt`，在本规格中说明用途、安装方式、打包方式和缺失时的降级行为。
- 先实现数据模型、JSON 解析、树构建和文本格式化，再实现 UI。
- 文本编辑和树视图之间通过路径索引同步，不要靠字符串模糊搜索定位节点。
- 树视图使用 Canvas 绘制，便于控制图标、点画线、选中背景和滚动性能。
- 右键菜单优先使用 Tk 原生 `Menu` 适配层，避免自绘级联菜单在不同平台上的焦点、悬停和关闭状态不稳定。
- 状态栏仅绘制状态或错误文本；固定版本和品牌信息统一放入顶部 `关于` 弹窗。
- 配置和自动保存失败必须被捕获，不能让应用崩溃。
- macOS 启动必须考虑 GUI Tk 需要 Python.app 入口的问题，普通命令行 Python 在某些环境中可能无法弹出窗口。
