# KindSee 问题记录与解决方案

本文档用于记录 KindSee 开发和验证过程中遇到的问题、原因、解决方案和后续预防措施。凡是经过多次试错才解决的问题，或者后续容易再次踩坑的问题，都应记录到这里。

新增记录时推荐格式：

```text

时间：
影响范围：
现象：
原因：
解决方案：
预防措施：
关联文件：
验证方式：
```

## 1. macOS 普通 Python 可能无法正常启动 Tk GUI

时间：初始开发阶段

影响范围：macOS 双击启动、GUI 冒烟测试

现象：

- 直接使用某些命令行 Python 启动 Tk 程序时，窗口可能无法正常弹出。
- macOS 对 Tk GUI 程序更推荐使用 `Python.app` 入口。

原因：

- macOS 上 Tk GUI 与普通命令行 Python 入口存在兼容差异。
- Homebrew 或官方 Python 安装器通常会提供 `Resources/Python.app/Contents/MacOS/Python`。

解决方案：

- `run_macos.command` 中优先查找 Python.app 入口。
- 找不到 Python.app 时再尝试 `python3.12`、`python3`。
- 启动失败时写入 `logs/macos-launch.log`。

预防措施：

- 后续不要把 macOS 启动脚本简化为只执行 `python3 kindsee.py`。
- 修改启动脚本后必须重新做 macOS GUI 冒烟验证。

关联文件：

- `run_macos.command`

验证方式：

```bash
./run_macos.command
```

## 2. macOS 自动读取窗口标题可能被辅助访问权限拦截

时间：初始 GUI 冒烟验证阶段

影响范围：自动化验证

现象：

- 使用 `osascript` 读取 Python 窗口标题时出现权限错误。
- 错误类似：`osascript 不允许辅助访问`。

原因：

- macOS 对 System Events 操作窗口信息需要辅助访问权限。
- 当前环境未授予相关权限。

解决方案：

- 不把窗口标题读取作为唯一 GUI 验证依据。
- 可通过启动脚本日志确认 Python.app 入口被调用。
- 必要时人工观察窗口是否弹出。

预防措施：

- 后续自动化验证不要默认依赖 macOS 辅助访问权限。
- 如需做完整 UI 自动化，应先确认权限或使用更适合的 GUI 测试工具。

关联文件：

- `run_macos.command`

验证方式：

```bash
tail -80 logs/macos-launch.log
```

## 3. 鼠标滚轮 delta 在 macOS 和 Windows 上不完全一致

时间：初始开发阶段

影响范围：文本编辑器滚动、树视图滚动、字体缩放

现象：

- 早期实现中直接使用 `int(event.delta / 120)`。
- 在部分 macOS 环境下，滚轮 delta 不一定稳定为 120 的倍数，可能导致滚动不灵敏或不滚动。

原因：

- Tk 在不同平台上返回的 `event.delta` 语义存在差异。

解决方案：

- 改为根据正负号判断滚动方向：

```python
units = -1 if event.delta > 0 else 1
```

预防措施：

- 后续涉及滚轮事件时，不要假设所有平台都是 120 的固定步长。
- 字体缩放、横向滚动、纵向滚动都应统一处理方向。

关联文件：

- `kindsee_app/widgets/common.py`
- `kindsee_app/widgets/tree_canvas.py`

验证方式：

- 文本区滚轮上下滚动。
- 树视图滚轮上下滚动。
- `Ctrl + 鼠标滚轮` 调整字体大小。
- `Shift + 鼠标滚轮` 横向滚动树视图。

## 4. 单文件实现导致 JSON 逻辑与应用控制器强耦合

时间：架构重构阶段

影响范围：后续支持 TXT、JS、Java 等文件类型

现象：

- 原始 `kindsee.py` 超过 1500 行。
- 配置、主题、JSON 解析、树构建、位置索引、UI 控件、页签、文件操作都混在一个文件中。
- 如果新增文件类型，容易在主窗口类中继续堆 `if/else`。

原因：

- 初始实现以快速满足 JSON 功能为主，没有先建立文档类型扩展边界。

解决方案：

- 拆分为 `kindsee_app/` 包。
- 抽出 `documents/base.py` 定义 `DocumentType` 和 `DocumentRegistry`。
- 把 JSON 专属逻辑迁移到 `documents/json_document.py`。
- `app.py` 只调用当前文档类型策略，不直接调用 `json.loads()`。

预防措施：

- 新增文件类型时必须新增文档类型策略，不要把具体解析逻辑写入 `app.py`。
- 如果 `app.py` 出现大量文件类型判断，说明架构开始退化。

关联文件：

- `kindsee.py`
- `kindsee_app/app.py`
- `kindsee_app/documents/base.py`
- `kindsee_app/documents/json_document.py`

验证方式：

```bash
python3.12 -B -m py_compile kindsee.py kindsee_app/*.py kindsee_app/documents/*.py kindsee_app/widgets/*.py
python3.12 -B -c "from kindsee_app.documents import default_registry; doc=default_registry().get('json'); result=doc.parse('{\"a\":1}'); assert not result.error; print('ok')"
```

## 5. 清空文本时状态文案曾写死为 JSON

时间：架构重构阶段

影响范围：后续多文件类型支持

现象：

- 清空文本后直接显示 `请输入 JSON 内容`。
- 对 TXT、JS、Java 等文件类型来说，该提示不一定合适。

原因：

- 应用层仍残留 JSON 专属文案。

解决方案：

- 清空文本后调用当前文档类型的 `parse()`。
- 由具体文档类型返回自己的空内容状态。

预防措施：

- 应用层不直接写具体文件类型的语法错误、空内容提示、格式化规则。
- 这些文案应尽量由文档类型策略返回。

关联文件：

- `kindsee_app/app.py`
- `kindsee_app/documents/json_document.py`

验证方式：

- 清空 JSON 文本后，状态栏应显示 JSON 文档类型提供的空内容提示。
- 后续新增文件类型时，应验证其空内容状态独立生效。

## 6. 主题不应由文档类型模块直接控制

时间：架构讨论阶段

影响范围：主题切换、多文件类型语法渲染

现象：

- 如果 JSON、JS、Java 等模块直接设置 Tk 控件颜色，后续主题切换会出现重复代码和不一致表现。

原因：

- 文档类型语义和 UI 渲染职责混淆。

解决方案：

- `theme.py` 只定义主题令牌。
- `widgets/` 根据主题令牌刷新控件。
- `documents/` 只返回结构、路径、语义 token，不直接操作 UI 颜色。
- `app.py` 在主题切换时统一调用控件的 `set_palette()`。

预防措施：

- 文档类型模块不得直接调用 `Text.tag_configure()` 设置颜色。
- 如需语法高亮，文档类型应返回类似 `("keyword", start, end)` 的语义结果，由通用编辑器根据主题渲染。

关联文件：

- `kindsee_app/theme.py`
- `kindsee_app/widgets/common.py`
- `kindsee_app/widgets/tree_canvas.py`
- `kindsee_app/documents/json_document.py`

验证方式：

- 切换明暗主题后，文本区、树视图、状态栏、菜单、滚动条颜色应同步变化。
- 文档类型模块中不应出现具体 UI 颜色控制。

## 7. 验证产生的临时文件需要清理

时间：开发验证阶段

影响范围：项目目录整洁度

现象：

- 运行 Python 编译或启动脚本后会产生：
  - `__pycache__/`
  - `kindsee_app/**/__pycache__/`
  - `logs/`

原因：

- Python 导入模块会生成字节码缓存。
- 启动脚本会写入日志文件。

解决方案：

- 验证完成后清理临时目录。

预防措施：

- 提交或交付前检查 `rg --files`。
- 如果未来引入版本控制，应加入 `.gitignore`。

关联文件：

- `__pycache__/`
- `kindsee_app/**/__pycache__/`
- `logs/`

验证方式：

```bash
rg --files
```

## 8. 页签使用普通 Tk 控件实现时偶发无响应或样式失控

时间：UI 优化阶段

影响范围：页签切换、关闭页签、新建页签、顶部工具栏

现象：

- 用户点击页签时偶发无法切换。
- 同一状态下点击页签关闭按钮 `×` 和新增页签 `+` 也可能无反应。
- 将页签和顶部菜单入口改为原生 `Button` 后，macOS 下仍出现明显边框，视觉不符合无边框工具栏和页签要求。

原因：

- 页签条使用 `Label` 加 `<Button-1>` 事件绑定实现点击。
- 页签切换、关闭、新建过程中会刷新并销毁重建页签控件，基于事件绑定的标签控件在复杂焦点/重绘时更容易出现点击事件未按预期触发。
- 原生 `Button` 的边框、内边距和高亮行为在不同平台主题下不可完全控制，尤其 macOS 下容易呈现传统按钮外观。
- 页签条是高频重绘区域，把每个页签拆成多个原生控件会增加控件重建和事件路由复杂度。
- 关闭当前活动页签后，如果 `active_tab_id` 仍指向已删除页签，后续新建、关闭或切换页签时会触发 `StopIteration`，表现为点击无响应。

解决方案：

- 顶部工具栏改为 `Canvas` 自绘图标入口，集中维护打开、保存、设置三个命中区域。
- 页签栏改为 `Canvas` 自绘页签、关闭区域和新增区域，集中维护命中测试。
- 点击事件只绑定在画布上，根据命中区域分派到切换、关闭、新建等动作。
- 切换、新建、关闭页签前先安全获取当前活动页签；关闭当前页签后先切换到仍存在的页签，避免活动页签 ID 指向已删除对象。
- 主题切换时重绘画布，避免原生控件边框和平台主题差异污染视觉。

预防措施：

- 页签、工具栏等需要无边框且跨平台一致的核心入口，优先使用集中式自绘和命中测试。
- 不要在页签条中为每个页签拆分多个短生命周期 Tk 控件。
- 如果新增工具栏入口，应在统一的绘制列表和命中区域列表中扩展，不要零散新增按钮。

关联文件：

- `kindsee_app/app.py`

验证方式：

- 连续点击不同页签，应能稳定切换。
- 连续点击 `×` 应能关闭页签。
- 连续点击 `+` 应能新增页签。

## 9. 右键菜单层级、同步开关和状态栏细节偏离预期

时间：右键菜单与同步显示优化阶段

影响范围：右键菜单、类型切换、文本编辑器、状态栏、顶部工具栏

现象：

- 右键菜单一级菜单直接平铺 `JSON 文件`、`TXT 文件` 等类型项，菜单层级不清晰。
- 类型项显示为完整类型名称，不如 `.json`、`.txt`、`.js` 等扩展名直观。
- 已有自动检测逻辑后，右键菜单中仍保留 `检测` 命令，造成入口冗余。
- 开启同步后，文本窗口插入光标和编辑体验异常。
- 单文本窗口类型仍可点击同步开关。
- 右键菜单切换 `同步` 后，鼠标回到文本编辑框时仍显示为指针光标，没有恢复为文本编辑光标。
- 状态栏右侧版本文字没有垂直居中。
- 顶部打开、保存图标视觉尺寸明显大于设置图标。
- 选中菜单项使用 `✓ 同步` 这种内联前缀后，选中项正文与未选中项正文无法左侧对齐。
- 鼠标放到右键菜单项上没有高亮，放到 `类型` 上没有触发二级菜单。
- 鼠标在一级菜单上移动时，右键菜单会自动消失。
- 鼠标从拥有二级菜单的 `类型` 移动到普通一级菜单项时，或从二级菜单移回一级菜单时，菜单仍会自动消失。
- 二级菜单打开后，鼠标移动到普通一级菜单项时，二级菜单没有自动关闭。
- 关闭一次右键菜单后，再次右键无法打开菜单。

原因：

- 早期自定义右键菜单只支持一层菜单，只能把类型项平铺。
- 菜单项复用了 `display_name`，没有针对右键菜单选择更简洁的扩展名展示。
- 同步切换与右键菜单弹出过程存在抢焦点行为，影响文本框焦点恢复。
- 同步功能没有按当前文档布局模式限制可用状态。
- 原生菜单关闭和同步命令执行后没有显式恢复文本控件 cursor，导致平台菜单的指针样式残留到文本编辑区域。
- 状态栏文字使用中心锚点绘制，在不同字体度量下视觉中心不稳定。
- 工具栏直接使用 emoji 图标，macOS 下不同 emoji 的实际字面尺寸不一致。
- 自定义菜单没有预留独立标记列，选中标记占用了正文列空间。
- 无边框 `Toplevel` 菜单只依赖 Canvas 的 `<Motion>` 事件；在 macOS 上某些弹出场景中移动事件不稳定，导致悬停状态和子菜单触发失效。
- 菜单增加了“指针离开后自动关闭”的定时逻辑；在无边框弹窗和多窗口菜单场景下，指针位置判断可能误判为离开菜单，从而销毁一级菜单。
- 二级菜单使用独立无边框 `Toplevel`，在一级和二级窗口之间移动时会产生复杂的进入/离开事件；如果同时在悬停变化中销毁子菜单窗口，容易连带触发父菜单异常关闭。
- 为避免根菜单误关，曾取消了悬停到普通一级项时关闭子菜单的逻辑，导致子菜单残留。
- 根菜单关闭时只在 `grab_current() is self` 条件下释放 grab，某些平台状态下判断不稳定，导致 grab 没有被释放，后续右键事件无法正常进入编辑器。

解决方案：

- 扩展 `FastContextMenu` 支持二级菜单；一级只显示 `类型`，二级显示扩展名列表。
- 移除右键菜单 `检测` 命令和对应入口代码，保留空白到非空、整段替换触发的自动检测。
- 右键类型子菜单统一使用文档类型默认扩展名展示，当前类型用 `✓` 标记。
- 右键菜单弹出时不再强制抢焦点；切换同步后把焦点还给文本编辑器。
- 文本编辑器使用平台文本光标；macOS 为 `ibeam`，Windows 和其它 Tk 平台为 `xterm`，同步切换、文本右键菜单关闭、鼠标重新进入文本框时只做低频幂等恢复。
- 单文本窗口类型下 `同步` 菜单项置灰不可用。
- 顶部打开、保存图标改为 Canvas 几何绘制，保证与设置图标尺寸一致。
- 状态栏按字体行高计算绘制 y 坐标，保证右侧文字垂直居中。
- 自定义菜单固定左侧标记列，`✓` 绘制在标记列中，正文统一从同一 x 坐标开始。
- 自定义菜单增加悬停状态字段和指针轮询；`<Motion>` 与定时检测当前鼠标位置共同驱动高亮和二级菜单展开。
- 移除右键菜单的自动关闭定时器；菜单关闭只由点击菜单项、按 Esc、打开下一次右键菜单等明确动作触发。
- 根菜单使用 Tk grab 接管菜单打开期间的点击事件，点击菜单区域外才关闭；菜单内移动只更新高亮和子菜单显示，不销毁父菜单。
- 悬停到普通一级菜单项时，只关闭当前子菜单，不关闭根菜单。
- 根菜单销毁时无条件尝试 `grab_release()`，确保后续右键菜单可以再次触发。
- 最终放弃自绘 `Toplevel + Canvas` 级联菜单，改用 Tk 原生 `Menu` 作为 `FastContextMenu` 的底层实现。
- `FastContextMenu` 只保留适配职责：主题配置、菜单项名称、二级菜单、置灰状态、checkbutton 选中标记和命令绑定。
- 原生 `Menu` 接管悬停高亮、二级菜单展开/收起、点击外部关闭、Esc 关闭和平台级菜单交互细节。

预防措施：

- 需要分类项时优先设计为级联菜单，不要把所有子项平铺到一级菜单。
- 右键菜单显示文案应服务操作效率，文件类型优先使用扩展名。
- 同步功能只能影响定位、滚动和高亮，不得改变文本编辑器的可编辑状态和焦点体验。
- 菜单命令执行后如果焦点回到文本编辑器，必须同步恢复文本编辑器鼠标光标样式。
- 自绘图标不要混用实际尺寸不可控的 emoji。
- 菜单选中、复选等状态应使用独立标记列表达，不要把状态符号拼进正文布局。
- 自绘弹出菜单不要只依赖单一鼠标移动事件；关键悬停行为应有当前指针位置校正。
- 不要用短周期自动关闭逻辑销毁右键菜单；如需点击外部关闭，应单独实现明确的外部点击检测。
- 级联菜单的生命周期必须由根菜单统一管理；悬停行为不能销毁根菜单。
- 关闭根菜单时必须释放所有临时输入抓取状态。
- 对平台交互细节复杂的基础控件，优先使用 Tk 原生控件；自绘只用于原生控件无法满足且行为边界简单的场景。

关联文件：

- `kindsee_app/app.py`
- `kindsee_app/widgets/common.py`
- `kindsee_app/widgets/status_bar.py`

验证方式：

- 文本/树右键菜单一级显示 `类型`，悬停后显示 `.json`、`.txt`、`.log`、`.xml`、`.java`、`.py`、`.js`。
- 右键菜单中不再出现 `检测`。
- JSON/XML 分栏类型可切换 `同步`，TXT/LOG/Java/Python/JavaScript 单窗口类型下 `同步` 置灰。
- 开启同步后文本编辑器仍能看到插入光标并正常输入。
- 切换 `同步` 后，鼠标位于文本编辑器时应显示文本编辑光标。
- 状态栏右侧版本文字垂直居中。
- 选中 `同步` 后，`同步` 文本应与 `复制`、`粘贴` 等菜单项左侧对齐。
- 鼠标悬停菜单项时应显示高亮，悬停 `类型` 时应显示二级类型菜单。
- 鼠标在一级菜单内移动时，菜单不应自动消失。
- 鼠标在 `类型` 与其它一级菜单项之间移动、在二级菜单与一级菜单之间移动时，根菜单都不应自动消失。
- 鼠标从 `类型` 移动到普通一级菜单项时，二级菜单应自动关闭。
- 关闭一次右键菜单后，应能再次右键打开菜单。

## 10. 结构树版本显示、连线锚点和类型图标留白问题

时间：结构树视觉细节优化阶段

影响范围：状态栏、JSON/XML 结构树

现象：

- 状态栏右下角版本显示为 `v2.1.0`，与当前规格版本不一致。
- 结构树父子节点点画线没有从父节点类型图标底部中点引出。
- 类型图标左右留白过大，尤其 `T`、`{ }`、`[ ]` 图标占位显得松散。
- 类型图标改为动态宽度后，同一层级不同节点类型的节点名称起始位置没有对齐。
- 右侧文本点击对象属性名时，左侧结构树只能定位到父节点，不能定位到对应子节点。
- `‹ ›` 字形视觉高度明显小于 `{ }`、`[ ]`，三类类型图标外框高度不统一。
- 节点图标左右内边距仍偏大，字符到边框的空白过宽。
- 节点图标两个符号之间使用完整空格，导致图标整体仍显得偏宽。
- 半空格间距和左右总 padding 仍偏宽，图标宽高比例还不够接近正方形。
- 点击左侧结构树节点时，右侧只高亮 value 或 `{`/`[`，输入光标仍停在原位置，没有移动到对应 key。
- 左右符号按文本起点绘制时，受字体字宽和取整影响，图标左右两侧留白不对称。
- 右键菜单切换同步后，鼠标回到文本编辑器区域仍可能残留为指针光标，而不是文本编辑光标。
- 单击左侧结构树节点时，右侧会先选中或高亮 value，再把输入光标移动到 key 附近；预期应直接定位输入光标，不先选中 value。
- 为了修复右键菜单后的光标残留，曾在文本控件 `<Motion>` 中反复恢复 cursor，导致鼠标移动时光标快速闪烁，并且 macOS 下文本区仍可能显示为指针。
- 第一次打开窗口时，如果鼠标已经位于文本编辑器上方，系统可能没有触发文本控件的进入事件，导致实际鼠标样式仍停留在指针；切换页签后由于重新进入文本区才恢复为文本光标。
- 文本右键菜单弹出并关闭后，实际鼠标样式可能停留在菜单使用的指针样式，即使文本控件 `cursor` 配置值仍然正确。

原因：

- 版本常量更新后没有同步回 `APP_VERSION_TEXT` 和规格说明。
- 树连线使用缩进位置估算锚点，没有基于父节点类型图标实际坐标计算。
- 类型图标使用统一固定宽度，普通字段图标和对象/数组图标都被迫占用较宽空间。
- 后续按单个图标文本动态计算宽度，虽然减少了留白，但破坏了同层节点文本对齐。
- JSON 位置索引只把值区域映射到节点；当光标落在属性名或冒号附近时，不在子节点值范围内，只能命中父对象范围。
- 图标外框高度依赖原有行内固定上下坐标，不是按统一图标高度集中计算。
- 图标宽度保留的左右 padding 仍偏大。
- 图标符号作为一个带空格的字符串绘制，无法单独控制中间空格的视觉宽度。
- 图标宽度只按符号宽度加固定 padding 计算，没有约束为接近图标高度。
- 结构树选中到文本的同步只调用高亮逻辑，没有移动 Text 控件的 `insert` 标记。
- 位置索引没有区分“高亮范围”和“光标落点”；对象属性需要把光标落点设为 key 起点，顶层节点需要落到 `{` 或 `[` 起点。
- 图标符号使用左锚点绘制，整体居中只保证数学宽度居中，不保证左右槽位视觉对称。
- 文本编辑器只在鼠标进入和同步切换后恢复 `cursor`，菜单命令执行后或鼠标在文本区移动过程中仍可能被平台菜单状态影响。
- 树节点选中复用了“高亮范围并移动光标”的组合方法，导致树到文本定位无法做到只移动输入光标。
- 对象属性光标落点记录在 key 开始双引号位置，根节点落点记录在 `{` 或 `[` 起点，不符合当前直接编辑的定位预期。
- 后续修复把“恢复光标”放到 `<Motion>` 高频事件中，本质是用事件补丁掩盖 cursor 归属不清的问题；频繁 `configure(cursor=...)` 会触发平台光标重绘，造成闪烁。
- `xterm` 是 Tk 默认文本光标名，但 macOS/Aqua 下更适合显式使用 `ibeam`，否则实际视觉效果可能不符合系统原生文本编辑光标预期。
- 窗口初次映射或原生菜单关闭后，Tk 不一定重新分发文本控件的进入事件；只检查 `Text.cget("cursor")` 会误判，因为配置值正确不代表系统当前显示的鼠标样式已经刷新。
- `tk_popup()` 后面的代码不应简单理解为“菜单关闭后一定执行”；原生菜单和平台事件顺序存在差异，需要允许菜单关闭后或用户回到文本区时再做一次低频刷新。

解决方案：

- 将 `APP_VERSION_TEXT` 调整为 `当前版本:v1.0.0`。
- 结构树绘制时先计算每个可见节点的类型图标位置，再根据父节点图标底部中点绘制父子连线。
- 普通值图标符号从 `T` 改为 `‹` 和 `›`。
- 类型图标宽度改为按 `{}`、`[]`、`‹›` 加半空格间距后的最大宽度统一占位，并只保留少量左右内边距。
- 三种节点类型分别使用浅色背景区分：对象浅红、数组浅黄、普通值浅绿；深浅顺序为浅红最深、浅黄次之、浅绿最淡。
- 对象、数组和普通值图标都改为左右符号分开绘制，不再依赖完整空格字符。
- JSON 对象属性建立从属性名起点到属性值结束的节点映射，让点击 key、冒号或 value 都能定位到对应子节点。
- 类型图标外框高度改为统一高度，并在行高中垂直居中绘制。
- 类型图标左右内边距进一步收窄，只保留字符到边框的少量可辨识空隙。
- 类型图标改为左右符号分开绘制，中间间距先使用半个空格宽度；统一占位宽度按符号宽度加间距计算。
- 进一步将符号间距压缩到约八分之一空格宽度，左右总 padding 压缩到约 1px。
- 图标宽度至少等于图标高度，让类型图标整体更接近正方形。
- JSON 位置索引增加 `cursor_positions`：对象属性记录 key 起点，顶层和数组元素记录值起点。
- 早期编辑器新增高亮并移动输入光标的同步方法，后续拆分为“高亮”和“只移动输入光标”两个独立能力。
- 图标左右符号改为以图标中心为基准的对称槽位绘制；左右符号分别使用 `center` 锚点放置。
- 文本编辑器新增只移动输入光标的方法，树节点单击改为只移动 `insert` 标记并清除临时同步高亮，不再先选中或高亮 value。
- 早期曾在鼠标移动、鼠标按下/释放等事件中显式恢复 `xterm` 文本编辑光标，后续废弃该高频恢复方式，改为平台文本光标和幂等恢复。
- JSON 根节点光标落点调整为顶层 `{` 或 `[` 右侧；对象属性节点光标落点调整为 key 开始双引号右侧；数组元素等没有 key 的节点仍落到值起点。
- 将文本光标名称抽取到 `platforms.text_edit_cursor_name()`：macOS 使用 `ibeam`，Windows 和其它 Tk 平台使用 `xterm`。
- 文本控件初始化时一次性设置平台文本光标；`restore_text_cursor()` 改为幂等方法，只在 cursor 被其它代码实际改掉时才重新配置。
- 移除文本控件 `<Motion>`、鼠标按下和释放中的反复 cursor 恢复逻辑，避免鼠标移动时快速闪烁。
- 新增 `request_text_cursor_refresh()`：窗口初次显示、页签切换、右键菜单弹出后只设置一次性刷新标记，并低频强制刷新一次。
- 文本控件保留 `<Enter>` 和 `<Motion>` 绑定，但只在存在刷新标记时执行一次强制刷新，随后立即清除标记；普通鼠标移动不再反复配置 cursor。
- 强制刷新时先短暂切到通用 `arrow`，再在 idle 阶段切回平台文本光标，用一次真实 cursor 变更促使 macOS/Tk 重新应用当前文本控件的鼠标样式。

预防措施：

- 修改版本号时必须同步代码常量、规格说明和状态栏验收项。
- 树结构连线不能用缩进常量硬编码，应基于实际图标几何信息绘制。
- 同一层级节点名称必须左侧对齐；图标占位可以统一，但应基于当前图标文本集合的最大宽度，而不是过宽的硬编码宽度。
- 文本 offset 到树节点 path 的映射应覆盖用户可点击的完整属性区域，不应只覆盖 value。
- 类型图标尺寸应由统一宽度、统一高度和少量内边距共同控制，不要让不同字形导致外框大小变化。
- 图标内部间距应独立控制，不要依赖普通空格字符的固定宽度。
- 类型图标需要同时约束内部间距和宽高比例，避免只收窄 padding 但仍显得过宽。
- 同步定位应分别处理高亮范围和输入光标位置，不要把二者混成同一个 offset。
- 图标内部两个符号不要用左锚点起始位置拼接，应使用中心对称的绘制方式保证左右留白一致。
- 树节点单击属于定位操作时，不应复用 value 高亮方法；需要把“文本选中/高亮”和“输入光标定位”拆成独立编辑器能力。
- 涉及原生菜单的命令回调后，如果焦点回到文本编辑器，必须在多个短延迟点恢复文本控件 cursor，抵消平台菜单状态残留。
- cursor 应有明确归属：文本控件负责自己的文本光标，工具栏、页签、状态栏只负责自身悬停光标，不能通过父窗口或高频事件反复覆盖。
- 不要在 `<Motion>` 事件中做 `configure(cursor=...)` 这类高频 UI 配置；确需恢复时应在进入控件、菜单命令结束或状态变化时做幂等恢复。
- 允许在 `<Motion>` 中做一次性 pending 检查，但不能每次移动都配置 cursor；检查和实际刷新必须通过状态标记隔离。

## 12. macOS 下追加选中相同内容高亮失效

时间：相同内容多查询高亮修复阶段

影响范围：文本编辑器、相同内容高亮、macOS 交互

现象：

- Windows/Linux 预期按住 Ctrl 选中文本可以把当前选中内容追加到已有相同内容高亮查询组。
- macOS 下按住 Ctrl 无法稳定进行追加选择。
- macOS 下按住 Command 拖拽可以临时选中另一段文本，但松开鼠标后，前面选中的内容会变成非选中状态，追加高亮没有按预期保留。
- macOS 下按住 Command 双击选中词时，选中的词没有稳定追加到已有相同内容高亮查询组。
- 追加选择后松开 Command/Ctrl，已有多个查询词的高亮会退化为只保留最后选中的内容。
- 进一步测试发现，松开 Command/Ctrl 时如果 Tk 仍保留最后一次真实选区，旧逻辑会把这次修饰键释放当作普通选区变化，导致查询组被最后一个选中文本替换。

原因：

- 旧实现只通过 `event.state & 0x0004` 判断 Ctrl，没有把 macOS 的 Command 修饰键纳入追加高亮触发条件。
- 旧实现为了防抖，在鼠标释放 120ms 后才读取 `sel.first` 和 `sel.last`；macOS/Tk 可能在这段时间内替换或清理当前选区，导致本次选中文本丢失。
- 相同内容高亮本质上不应依赖 Tk 保留多个真实选区，而应把每次选中的文本保存为高亮查询词。
- 松开 Command/Ctrl 会触发 `<KeyRelease>`，旧逻辑在没有当前选区时会清空 `occurrence_queries`，导致已经追加的查询词被删除。
- Command/Ctrl + 双击会先经过一次单击释放；如果这次释放暂时没有形成选区，旧逻辑可能先清空查询组，随后双击选中的词只能作为新的最后一组存在。
- 前一次修复只覆盖了“修饰键释放时没有选区”的情况，没有识别“释放的是修饰键本身”。当释放修饰键时仍存在最后选区，代码仍会按普通选择路径替换整个查询组。
- `keep_query=True` 的语义不严格：虽然调用方意图是保留已有查询组，但回调仍会读取当前 Tk 选区；当最后一次真实选区仍存在时，保留路径又退回成了普通替换路径。

解决方案：

- 追加高亮修饰键按平台识别：Windows/Linux 使用 Ctrl，macOS 使用 Command，并兼容识别 macOS Ctrl。
- 在鼠标释放或键盘选区变化时立即读取当前选中文本，保存为待高亮查询词，再执行防抖高亮。
- 防抖回调优先使用已捕获的待高亮文本；只有没有待高亮文本时才读取当前 Tk 选区。
- 多个查询词继续使用 `occurrence_queries` 持久保存，不依赖 Tk 的真实多选区。
- `<KeyRelease>` 没有当前选区时不清空已有查询组，只保留并重绘已有高亮。
- 追加操作没有当前选区时不清空已有查询组，避免 Command/Ctrl + 双击的第一次单击释放破坏历史查询词。
- Command/Ctrl + 双击最终形成选区后，按追加高亮逻辑把双击选中的词加入查询组。
- 明确识别 Ctrl/Command/Meta/Super 等追加选择修饰键的 `<KeyRelease>`；释放这些修饰键时，无论当前是否还有最后一次真实选区，都只保留并重绘已有查询组，不执行普通选区替换。
- 收紧 `keep_query=True` 语义：保留查询组时绝不读取当前 Tk 选区，只按已有 `occurrence_queries` 重绘高亮。

预防措施：

- macOS 的 Command/Ctrl/Option 修饰键不能直接套用 Windows 的 Ctrl 判断，涉及跨平台快捷键或鼠标修饰键时必须做平台化处理。
- 需要防抖的选区功能，应在事件发生时捕获关键数据，防抖只延迟计算，不延迟读取易变化的 UI 状态。
- 不要把“真实多选区”和“多个高亮查询词”混为一谈；Tk `Text` 默认只有一个主要选择区，多查询高亮应由应用状态维护。
- 修饰键释放、空选区的追加事件、双击前置单击释放都不应被当作“用户明确要求清空高亮”的操作。
- 修饰键释放必须先按 `keysym` 判断事件性质，不能只根据当前是否有选区决定是否替换查询组。
- 表示“保留查询组”的参数必须真正隔离当前 UI 选区状态，不能在内部再读取 `sel.first/sel.last`。

关联文件：

- `kindsee_app/widgets/common.py`
- `docs/KindSee-v1.0.0-软件开发规格说明.md`

验证方式：

- Windows/Linux 按住 Ctrl 选中文本时，追加到已有相同内容高亮查询组。
- macOS 按住 Command 选中文本时，追加到已有相同内容高亮查询组。
- macOS 按住 Command 双击选中文本时，追加到已有相同内容高亮查询组。

## 13. 顶部工具栏图标审美和风格不统一

时间：顶部工具栏视觉优化阶段

影响范围：顶部工具栏、打开按钮、保存按钮、设置按钮

现象：

- 当前打开、保存图标使用简单几何草图，外形生硬，缺少细节和层次。
- 打开、保存图标与设置图标风格不一致，视觉重量不统一。

- 打开图标按参考图进一步重绘为前方黄橙色矩形文件夹叠加后方蓝边白色文档，移除此前不符合参考图的斜切底边文件夹形状。

## 14. SQL 内容被 Python 自动检测误判且 SQL 校验过浅

时间：2026-05-24

影响范围：SQL 自动检测、SQL 实时语法检查、SQL 右键格式化

现象：

- 输入 `select * from abc_t where aa = 1 and bb = c;` 时曾被误判为 Python，并显示 Python `invalid syntax`。
- 输入 `select * from1 abc_t where aa = 1 and bb = c;` 时曾不报 SQL 语法错误。
- 输入 `select * from abc_t where aa = 1a and1 bb = c;` 时曾不报 SQL 语法错误。
- SQL 页签右键菜单中的 `格式化` 曾处于灰色不可用状态。

原因：

- Python 内容自动检测规则把任意 `from <标识符>` 当作 Python 特征，误伤 SQL 的 `FROM 表名`。
- 初始 SQL 文档类型只检查括号、注释和字符串闭合，没有引入成熟 SQL 解析器，常见语句结构和 WHERE 表达式覆盖不足。
- SQL 文档类型的 `supports_format` 被设置为 `False`，没有提供 SQL 格式化策略。

解决方案：

- Python 内容检测收窄为真正的 `from ... import ...`。
- SQL 文档类型增加 `SqlGlotSyntaxValidator` 第三方解析器适配组件，优先使用 `sqlglot` 做 SQL 解析和格式化；依赖缺失时降级到 `BasicSqlSyntaxValidator`，覆盖 `from1`、`1a`、`and1`、缺少运算符、缺少操作数等基础错误，避免应用无法启动。
- SQL 文档类型启用 `supports_format = True`，并提供基础 SQL 美化输出。

预防措施：

- 新增语言内容检测时，规则必须尽量使用该语言的完整特征，不要用容易与其它语言冲突的孤立关键字。
- SQL 这类非标准库可完整解析的语言，如需更接近 JSON 的解析器驱动校验，应优先引入成熟解析库，并在依赖文件、规格说明和打包说明中明确声明依赖用途、默认方言和降级行为。
- 文档类型的菜单能力声明必须和实际策略一致，支持格式化时必须实现 `format_text()`。

关联文件：

- `kindsee_app/documents/python_document.py`
- `kindsee_app/documents/sql_document.py`
- `requirements.txt`
- `docs/KindSee-v1.0.0-软件开发规格说明.md`

验证方式：

```bash
python3.12 -B -c "from kindsee_app.documents import default_registry; doc=default_registry().get('sql'); assert not doc.parse('select * from abc_t where aa = 1 and bb = c;').error; assert doc.parse('select * from1 abc_t where aa = 1 and bb = c;').error; assert doc.parse('select * from abc_t where aa = 1a and1 bb = c;').error; assert doc.supports_format; print(doc.format_text('select * from abc_t where aa = 1 and bb = c;'))"
```
