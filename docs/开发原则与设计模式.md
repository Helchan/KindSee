# KindEdit 开发原则与设计模式

本文档用于约束 KindEdit 后续开发、重构和功能扩展。所有代码修改应优先遵循本文档，避免功能扩展后出现模块职责混乱、重复实现、跨模块强耦合和难以维护的问题。

## 1. 总体目标

KindEdit 的长期目标不是只支持 JSON，而是逐步扩展为可支持多种文本类文件的轻量查看、编辑、结构浏览和语法渲染工具。

后续可能支持的文件类型包括：

- JSON
- TXT
- JavaScript
- Java
- XML
- YAML
- Properties
- Log

因此，程序结构必须围绕“应用壳层 + 通用 UI + 文档类型扩展”组织。

## 2. 核心分层原则

当前推荐分层如下：

```text
kindedit.py
启动入口，不承载业务逻辑

kindedit_app/app.py
应用控制器，负责窗口、页签、文件、快捷键、会话、主题切换、文档类型调度

kindedit_app/config.py
配置和持久化

kindedit_app/theme.py
主题令牌和明暗主题选择

kindedit_app/platforms.py
平台差异封装

kindedit_app/tabs.py
页签数据模型

kindedit_app/widgets/
通用 UI 控件

kindedit_app/documents/
不同文件类型的解析、格式化、结构构建、复制、位置索引等能力

kindedit_app/syntax/
不同文件类型的语法 token 识别和语法渲染策略注册
```

## 3. 单一职责原则

每个模块只负责一个清晰职责。

必须避免：

- 在 `app.py` 中直接写某种文件类型的解析逻辑。
- 在 `documents/` 文档类型模块中直接操作 Tk 控件颜色、布局、菜单。
- 在 `widgets/` 通用控件中写死 JSON、JavaScript、Java 等具体文件类型规则。
- 在 `documents/` 文档类型模块中直接设置语法高亮颜色。
- 在 `syntax/` 语法模块中直接操作 Tk 控件。
- 在 `theme.py` 中直接调用具体控件。
- 在 `config.py` 中处理 UI 或文档解析逻辑。

推荐职责：

- `app.py`：调度。
- `documents/`：语义。
- `syntax/`：语法 token 识别。
- `widgets/`：渲染。
- `theme.py`：主题令牌。
- `config.py`：配置读写。
- `platforms.py`：平台能力。

## 3.1 代码质量与可扩展性原则

KindEdit 必须长期保持低耦合、高内聚、可扩展、可维护，并尽量符合开闭原则：对扩展开放，对无关既有行为修改保持克制。

必须遵循：

- 单一职责：一个模块、类或函数只承担清晰职责。
- 依赖倒置：应用层依赖稳定接口和策略对象，不直接依赖具体文件类型实现细节。
- 开闭原则：新增文件类型、语法渲染、结构视图、设置项时，优先通过注册表、策略类、通用控件扩展，而不是在主流程中堆条件分支。
- 高内聚：同一能力的状态、配置、行为和验证应集中在同一模块或清晰边界内。
- 低耦合：模块之间只通过明确接口通信，避免跨层直接读写内部状态。
- 行为兼容：重构必须保证原有功能、文件名展示、页签规则、用户配置、快捷键和跨平台启动不受影响。
- 可验证：结构性调整必须配套编译验证、必要的轻量行为验证或 GUI 行为验证。

可以做：

- 为了支撑后续扩展，在必要时拆分过大的模块、抽取公共接口、引入注册表、策略类、适配器或通用控件。
- 在保证原有功能完全不受影响的前提下，对职责混乱、重复代码、跨层依赖、难以验证的实现做必要重构。
- 把重复出现的 UI 行为、平台差异、配置读写、语法高亮、文档类型能力抽成公共模块。
- 为降低风险，把重构拆成小步骤：先补验证，再调整结构，再迁移调用点，最后清理旧逻辑。

不建议做：

- 为了快速实现功能，在 `app.py` 中继续增加大量具体文件类型判断。
- 在一个函数里同时处理 UI 绘制、业务解析、配置读写、平台兼容和状态持久化。
- 为了修复局部问题复制粘贴相似逻辑，导致多个位置各自维护同一规则。
- 在没有验证的情况下做大范围重命名、移动或抽象。
- 为了短期省事把临时状态、全局变量或硬编码常量散落到多个模块。

禁止做：

- 禁止让文档类型模块直接操作 Tk 控件、主题颜色或具体菜单。
- 禁止让语法渲染模块直接修改文本控件或主题。
- 禁止让通用控件写死 JSON、XML、Java 等具体文件类型规则。
- 禁止绕过注册表直接在主控制器里硬编码新增文件类型流程。
- 禁止在未确认影响范围和未做验证的情况下进行破坏性重构。
- 禁止为掩盖设计问题不断叠加特殊分支、事件补丁、状态补丁和平台硬编码。
- 禁止为了通过单一场景牺牲已有功能、跨平台行为或后续扩展能力。

当功能扩展导致现有结构开始变得难以维护时，必要重构是开发职责的一部分，不应等到代码明显混乱后再处理。重构必须以“不改变用户可见行为”为默认前提；如果确实需要改变行为，必须先同步规格说明并记录原因。

## 4. 文档类型扩展原则

新增文件类型时，应优先新增一个文档类型模块，而不是修改主控制器的大量逻辑。

例如新增 JavaScript 支持时，推荐新增：

```text
kindedit_app/documents/javascript_document.py
```

并实现统一接口：

```python
class JavaScriptDocumentType:
    type_id = "javascript"
    display_name = "JavaScript 文件"
    default_extension = ".js"
    file_dialog_patterns = ("*.js",)
    view_mode = "split"
    parse_on_change = True
    supports_format = True
    supports_compact = False
    empty_status = "请输入 JavaScript 内容"

    def matches_path(self, path): ...
    def parse(self, text): ...
    def format_text(self, text): ...
    def compact_text(self, text): ...
    def copy_node(self, node): ...
    def copy_node_value(self, node): ...
    def copy_node_path(self, node): ...
```

然后在文档类型注册表中注册。

新增文件类型时应做到：

- 不破坏 JSON 现有行为。
- 不修改应用标题、页签标题规则、启动脚本文件名。
- 尽量不修改通用 UI 控件，除非新增的是通用能力。
- 文件类型模块输出结构化语义，UI 控件根据主题和语义负责渲染。
- 根据文件类型声明合适的 `view_mode`，例如 JSON、XML 等层级格式使用左右分栏，TXT、LOG 等纯文本或大文本格式使用单文本窗格。
- 大文本文件类型应优先设置 `parse_on_change = False`，避免每次输入都读取全文或重建结构。

## 5. 设计模式

### 5.1 策略模式

不同文件类型使用不同策略对象。

JSON、TXT、JS、Java 等文件类型应分别实现自己的 `DocumentType`。

应用层不关心具体文件类型如何解析，只调用统一方法：

```python
doc_type.parse(text)
doc_type.format_text(text)
doc_type.compact_text(text)
doc_type.copy_node(node)
```

文档类型还应声明自身 UI 和性能特征：

```python
doc_type.view_mode
doc_type.parse_on_change
doc_type.supports_format
doc_type.supports_compact
doc_type.empty_status
```

### 5.2 注册表模式

使用 `DocumentRegistry` 管理所有文档类型。

应用打开文件时，根据文件扩展名识别文档类型：

```python
doc_type = registry.detect(path)
```

这样新增文件类型只需要注册新策略，而不是在 `app.py` 中不断增加 `if suffix == ...`。

### 5.3 组合优先

通用 UI 能力通过组合使用：

- `LineNumberText` 负责文本编辑和行号。
- `JsonTreeCanvas` 负责树结构显示。
- `StatusBar` 负责底部状态栏。
- `FastContextMenu` 负责右键菜单。
- `SlimScrollbar` 负责滚动条。

不要把所有 UI 行为重新合并到一个巨大的窗口类中。

### 5.4 依赖倒置

`app.py` 应依赖抽象能力，而不是具体 JSON 实现。

推荐：

```python
self.current_document_type().parse(text)
```

避免：

```python
json.loads(text)
```

如果 `app.py` 中出现大量具体文件类型逻辑，说明架构边界已经开始退化。

## 6. 主题系统原则

`theme.py` 是公共主题模块，但只负责定义主题令牌，不直接控制具体控件。

推荐关系：

```text
theme.py
定义颜色、字体、状态色、语法色等 token

widgets/
根据 token 刷新控件样式

documents/
返回语义信息，不直接处理颜色

app.py
主题切换时通知所有 UI 控件刷新
```

例如文档模块可以返回语法语义：

```python
("keyword", start, end)
("string", start, end)
("number", start, end)
```

具体颜色由通用编辑器根据当前主题决定：

```python
palette["syntax_key"]
palette["syntax_string"]
palette["syntax_number"]
```

禁止在文档类型模块中直接写：

```python
text.tag_configure("keyword", foreground="#0a66c2")
```

## 6.1 语法渲染原则

语法渲染必须通过独立的 `syntax/` 模块统一管理。

推荐结构：

```text
kindedit_app/syntax/base.py
定义 SyntaxToken、SyntaxHighlighter、SyntaxRegistry

kindedit_app/syntax/json_highlighter.py
JSON 语法 token 识别

kindedit_app/syntax/xml_highlighter.py
XML 语法 token 识别

kindedit_app/syntax/java_highlighter.py
Java 语法 token 识别

kindedit_app/syntax/python_highlighter.py
Python 语法 token 识别

kindedit_app/syntax/javascript_highlighter.py
JavaScript 语法 token 识别

kindedit_app/syntax/text_highlighter.py 或 NoopHighlighter
TXT 空高亮实现
```

语法模块只负责识别 token，例如：

```python
SyntaxToken(kind="key", start=0, end=6)
SyntaxToken(kind="string", start=9, end=16)
SyntaxToken(kind="number", start=20, end=23)
```

语法模块不得：

- 直接访问 Tk `Text` 控件。
- 直接设置颜色。
- 直接控制主题。
- 直接修改文档内容。

语法 token 的颜色由 `theme.py` 定义，由 `widgets/` 中的文本编辑控件应用。

新增语法渲染类型时，应新增对应 highlighter，并在语法注册表中注册，不应把语法扫描逻辑写入 `app.py`。

大文本语法渲染必须有保护策略，例如最大渲染字符数、最大 token 数、防抖或按需渲染，避免每次输入造成明显卡顿。

## 7. 配置系统原则

配置读写统一放在 `config.py`。

新增配置项时需要：

- 在 `AppConfig` 中定义默认值。
- 在 `load_config()` 中兼容旧配置。
- 在 `save_config()` 中写入新字段。
- 配置读取失败必须静默降级到默认值。
- 配置保存失败不得影响主功能。

不要在业务模块中直接拼配置路径或直接读写 `settings.json`。

## 8. UI 控件原则

通用控件应保持文件类型无关。

例如：

- 文本编辑器不应该知道 JSON。
- 树控件不应该知道 JSON 解析方式。
- 状态栏不应该知道文档内容语义。
- 右键菜单可以由应用层组装，但菜单组件本身不应知道业务命令。

控件应提供清晰方法：

```python
set_palette(palette)
set_font_size(size)
set_tree(root)
highlight_span(start, end)
```

所有文本类型通用的编辑器交互，例如行号、滚动条、选区行为、相同内容高亮、Ctrl 多选相同内容高亮，应优先放在 `LineNumberText` 等通用控件中实现。除非功能只属于某一种文件类型，否则不要写入具体文档类型模块。

### 8.1 鼠标样式原则

鼠标样式只能按以下优先级设置：

1. 超链接区域使用手形光标。
2. 文本编辑框内使用文本编辑光标。
3. 其它所有区域使用普通指针。

菜单按钮、工具栏按钮、页签标题、页签关闭按钮、页签新增按钮、树节点、普通状态栏文字等可点击但不是超链接的区域，不得使用手形光标。

文本编辑光标由文本编辑控件统一维护；超链接手形光标由对应控件在命中超链接区域时维护。不要在应用层散落设置特殊鼠标样式。

## 9. 文件与页签原则

页签应记录：

- `id`
- `title`
- `file_path`
- `autosave_path`
- `dirty`
- `content`
- `document_type`
- `large_content`
- `content_loaded`
- `large_source_path`
- `content_size`

大文本页签不得把完整内容长期放入 `TabState.content` 或配置文件。`content` 只用于普通小文本；大文本页签应通过 `large_source_path` 指向外部文件或内部快照文件，并用 `content_loaded` 表示当前编辑器中是否已加载完整内容。

文件名展示规则不得随意改动：

- 应用默认标题：`KindEdit`
- 打开文件或切换页签后：`KindEdit - <页签标题或文件名>`
- 已修改页签标题追加 ` *`

## 9.1 大文本处理原则

大文本能力必须作为架构级保护能力处理，不能通过在单个事件回调里临时加特殊判断解决。

必须遵循：

- 使用统一的大文本策略阈值，避免多个模块各自硬编码大小判断。
- 文件打开、剪贴板粘贴、自动保存和保存到文件必须支持分块流程，单次 UI 事件不得读写 200M 级完整字符串。
- 大文本页签进入保护模式后，应暂停全量解析、结构树构建、语法渲染、Markdown 预览、全文格式化、全文压缩和全文复制。
- 大文本编辑期间，应用层只能记录元数据、脏状态、来源文件和快照路径，不得每次按键调用 `Text.get("1.0", "end-1c")`。
- 通用文本控件负责分块插入、分块读取和可见区域相关渲染；`documents/` 和 `syntax/` 模块不得直接操作 Tk 控件。
- 小文本路径必须保持原有解析、语法渲染、结构视图、自动检测、格式化、压缩和复制行为。

不建议做：

- 在 `app.py` 里把 200M 文件读成一个字符串再交给控件。
- 在每次文本变化、页签切换、配置保存或自动保存时复制整段大文本。
- 让大文本 JSON、XML、Markdown 或代码文件继续执行全量解析和全量语法渲染。

## 10. 错误处理原则

用户输入、文件 IO、配置读写、自动保存、平台能力调用都必须防御性处理。

必须保证：

- 配置损坏不影响启动。
- 自动保存失败不影响编辑。
- 文件打开失败只在状态栏提示。
- 文件保存失败只在状态栏提示。
- 平台能力不可用时静默降级。
- 某个文件类型解析失败时不影响其它页签。

## 10.1 反复修复止损原则

同一个问题如果已经连续修复两次仍未解决，必须暂停继续叠加补丁，先重新审视原本实现方案和设计边界。

必须检查：

- 现有实现方案是否从根上不适合当前问题。
- 模块职责是否混乱，是否把平台差异、控件行为、业务逻辑混在一起。
- 是否应该抽取公共能力、调整接口、替换组件或改用更稳定的底层实现。
- 是否存在只修表象、不修根因的事件补丁、状态补丁或硬编码补丁。
- 是否缺少可复现验证，导致每次修改都靠猜测。
- 是否需要先写最小验证脚本或 GUI 行为验证，再继续修改代码。

处理要求：

- 第三次修改前必须说明新的根因判断和替代方案。
- 能通过架构调整解决的，不要继续在原方案上堆特殊分支。
- 能用平台原生稳定能力解决的，不要继续维护复杂且脆弱的自绘或模拟逻辑。
- 修复后应把问题、原因、最终方案和预防措施记录到 `docs/问题记录与解决方案.md`。
- 如果修复涉及功能行为变化，必须同步更新软件开发规格说明文档。

## 11. 修改前检查清单

每次新增功能或修改代码前，先确认：

- 这是应用层能力，还是文档类型能力？
- 这是通用 UI 能力，还是某种文件类型专属能力？
- 是否需要新增策略类，而不是修改 `app.py`？
- 是否需要新增或调整文档类型的 `view_mode`、`parse_on_change`、格式化能力声明？
- 是否需要新增或调整内容自动检测规则，并保证用户手动指定类型后不被自动覆盖？
- 是否需要让文本编辑器右键菜单自动使用文档类型注册表，而不是写死类型列表？
- 是否符合低耦合、高内聚和开闭原则，是否需要先做小范围重构再扩展功能？
- 是否涉及大文本性能，是否避免了每次按键读取全文或重建结构？
- 是否会破坏现有 JSON 行为？
- 是否会破坏 macOS / Windows 启动脚本？
- 是否需要新增配置项？
- 是否需要新增或调整设置窗口中的配置入口？
- 如果是同一问题的第三次修复，是否已经重新评估实现方案、设计边界和验证方式？
- 是否需要同步更新软件开发规格说明文档？
- 是否需要更新问题记录文档？

## 12. 规格说明同步原则

每次新增功能、调整功能、修改交互、改变配置字段、改变文件结构、改变启动或打包方式时，都必须同步更新软件开发规格说明文档。

当前规格说明文档路径：

```text
docs/KindEdit-v1.0.0-软件开发规格说明.md
```

必须同步更新的情况包括：

- 新增文件类型支持。
- 修改现有 JSON 行为。
- 修改 UI 布局、菜单、状态栏、快捷键。
- 新增或删除配置字段。
- 修改主题令牌或主题切换行为。
- 修改启动脚本或打包脚本。
- 修改自动保存、会话恢复、文件打开保存逻辑。
- 调整模块结构、扩展接口或文档类型注册机制。

代码行为和规格说明不一致时，应视为问题。后续维护时必须同时修正代码或规格说明，不能长期保留偏差。

## 13. 验证要求
